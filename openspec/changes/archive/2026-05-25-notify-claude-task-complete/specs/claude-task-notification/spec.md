## ADDED Requirements

### Requirement: Claude Code セッションへの識別子注入

Roola SHALL Claude Code を起動する PTY セッションに、当該セッションを一意に指す
タブ識別子 `ROOLA_TAB_ID` と、アプリ起動ごとに生成するランダムトークン
`ROOLA_NOTIFY_TOKEN` を環境変数として注入しなければならない。

#### Scenario: Claude Code セッション起動時に識別子を注入する

- **WHEN** ユーザーが `ClaudeSkillAction` を伴うランチャーエントリを起動する
- **THEN** その PTY プロセスの環境変数に `ROOLA_TAB_ID`（当該セッションを一意に指す
  値）と `ROOLA_NOTIFY_TOKEN`（当該アプリ起動のトークン）が設定される

#### Scenario: 非 Claude セッションには通知用識別子を要しない

- **WHEN** ユーザーが素のシェルまたは任意コマンドのエントリを起動する
- **THEN** タスク完了通知の対象外であり、起動は通知機能の影響を受けない

### Requirement: ローカル通知受信口

Roola SHALL アプリ内にローカル HTTP 受信口を起動し、ループバックアドレス
（127.0.0.1）のみにバインドしなければならない。受信口は Stop フックからの POST を
単一エンドポイントで受け付ける。

#### Scenario: ループバックのみにバインドする

- **WHEN** 受信口が起動する
- **THEN** 127.0.0.1（ループバック）にのみバインドされ、LAN/外部インターフェースには
  公開されない

#### Scenario: 既定ポートが使用中のときフォールバックする

- **WHEN** 既定ポートが既に使用されている
- **THEN** 受信口は空きポートにフォールバックして起動し、確定したポート番号を保持する

#### Scenario: フック完了通知の POST を受け付ける

- **WHEN** `tab_id` / `token` / `session_id` / `cwd` を含む JSON が受信口へ POST
  される
- **THEN** 受信口は内容を解釈し、照合処理へ引き渡す

### Requirement: セッション照合とトークン検証

Roola SHALL 受信した完了通知について、トークンが現在のアプリ起動のものと一致し、
かつ `tab_id` が現在有効なセッションに存在する場合に限り通知を発射しなければ
ならない。いずれかが不一致の場合は通知を発射してはならない（MUST NOT）。

#### Scenario: 有効な Roola セッションの完了で通知する

- **WHEN** 受信した `token` が現在のアプリ起動のトークンと一致し、`tab_id` が有効な
  セッションに存在する
- **THEN** macOS ローカル通知を発射する

#### Scenario: トークン不一致の POST を無視する

- **WHEN** 受信した `token` が現在のアプリ起動のトークンと一致しない
- **THEN** 通知を発射せず、リクエストを無視する

#### Scenario: 未知のタブ ID を無視する

- **WHEN** 受信した `tab_id` が現在有効なセッションに存在しない（既に閉じられた等）
- **THEN** 通知を発射しない

### Requirement: macOS ローカル通知の発射

Roola SHALL 照合に成功した完了通知について、macOS のローカル通知を発射しなければ
ならない。通知本文には対象セッションを識別できる情報（タブ名または作業ディレクトリ）
を含める。

#### Scenario: タスク完了を通知する

- **WHEN** 有効なセッションのターン完了が照合された
- **THEN** macOS 通知センターに通知が表示され、本文に対象セッションのタブ名または
  作業ディレクトリが含まれる

#### Scenario: 通知が未許可のとき発射しない

- **WHEN** ユーザーが macOS の通知許可を与えていない
- **THEN** 通知は表示されず、設定画面で未許可状態が分かる

### Requirement: 通知許可の取得

Roola SHALL 通知機能の利用にあたり macOS の通知許可をユーザーに要求し、現在の許可
状態を照会できなければならない。

#### Scenario: 初回に許可を要求する

- **WHEN** 通知機能が初めて有効化される
- **THEN** macOS の通知許可ダイアログが表示される

#### Scenario: 許可状態を照会する

- **WHEN** 設定画面が表示される
- **THEN** 現在の通知許可状態（許可 / 未許可）が取得され表示される

### Requirement: 通知機能の設定 UI

Roola SHALL 設定画面に Claude Code タスク完了通知のセクションを提供し、機能の
ON/OFF、使用中ポート、コピペ用のフック設定 JSON スニペット、通知許可状態と再許可
への導線を表示しなければならない。UI は Polaris デザインシステムに準拠する。

#### Scenario: 機能を ON/OFF する

- **WHEN** ユーザーが設定画面で通知機能のトグルを操作する
- **THEN** 機能の有効/無効が切り替わり、無効時は通知を発射しない

#### Scenario: フック設定スニペットを提示する

- **WHEN** ユーザーが設定画面の通知セクションを開く
- **THEN** 確定したポート番号が埋め込まれた、`~/.claude/settings.json` 向けの Stop
  フック JSON スニペットがコピー可能な形で表示される

#### Scenario: スニペットがトークンを環境変数参照にする

- **WHEN** スニペットが生成される
- **THEN** スニペットはポート番号を埋め込む一方、トークンは環境変数
  `$ROOLA_NOTIFY_TOKEN` を参照する形であり、アプリ再起動でトークンが変わっても
  ユーザーが貼り直す必要がない

#### Scenario: 未許可時に許可導線を示す

- **WHEN** 通知が未許可の状態で設定画面を開く
- **THEN** 未許可である旨と、許可を行う（または macOS のシステム設定へ誘導する）
  導線が表示される
