# Spec: polaris-design-system

Polaris デザインシステムの要件。思想は機能主義（装飾を排し、機能・精密さ・
情報密度で形を決める）。背景と判断理由は `../../design.md`、視覚リファレンスは
`lib/dev/cockpit_preview.dart`（第 17 稿）を参照。

## ADDED Requirements

### Requirement: デザイントークンの一元管理

Polaris の全デザイン値（色・トーン階層・角丸・グリッド単位・タイポグラフィ）は `PolarisTokens` という単一の `ThemeExtension` に集約され、UI コンポーネントはそこを経由して値を参照しなければならない（MUST）。色・角丸・余白をコンポーネント側にハードコードしてはならない（MUST NOT）。

#### Scenario: コンポーネントがデザイン値を参照する

- **WHEN** UI コンポーネントが色・角丸・余白・テキストスタイルを必要とする
- **THEN** `Theme.of(context)` 経由で `PolarisTokens` または `ThemeData` の
  対応する値を参照し、リテラルの `Color(0x...)` やマジックナンバーを直接
  記述しない

### Requirement: ダーク専用テーマ

システムはダーク専用の Polaris テーマのみを提供しなければならない（MUST）。
ライトテーマを提供してはならない（MUST NOT）。

#### Scenario: アプリ起動時のテーマ

- **WHEN** アプリが起動する
- **THEN** ダークの Polaris テーマが適用され、ライト/ダークの切替手段は
  存在しない

### Requirement: グラファイトの 2 トーン階層

地は純黒でなくグラファイト（僅かにリフトした暗色）とし、`well`（最も暗い＝ファイル一覧の沈んだ面）と `bg`（一段明るい＝筐体の枠）の 2 トーンで面を分けなければならない（MUST）。面の分離に drop shadow / elevation を使ってはならない（MUST NOT）。区切りはトーン差と 1px ラインのみで表現する。

#### Scenario: ファイル一覧と筐体の分離

- **WHEN** ファイル一覧領域（well）とトップバー/サイドバー/ステータスバー
  （bg）が同時に表示される
- **THEN** well が bg より暗いトーンで描画され、両者の境界は 1px ラインで
  示され、影は一切描画されない

#### Scenario: 計器ディスプレイパネル

- **WHEN** ファイル一覧が描画される
- **THEN** ファイル一覧は筐体（bg）から僅かにインセットされ、R=4px・1px
  ボーダーの「計器ディスプレイパネル」として well トーンで嵌め込まれる

### Requirement: 単一アクセント（ゴールド／アイスブルー切替）

アクセントは常に 1 色のみ運用しなければならない（MUST）。同時に複数のアクセント
色を併用してはならない（MUST NOT）。アクセント色はユーザー設定で暖色ゴールド
（デフォルト）と寒色アイスブルーから選択でき、選択中の 1 色を選択・現在地・
フォルダなど機能的に意味のある箇所に限定して使う。

#### Scenario: アクセントの適用箇所

- **WHEN** 行が選択される、または現在地・フォルダ・アクティブな操作対象が
  表示される
- **THEN** 選択中の単一アクセント色で点灯し、それ以外の領域は無彩色の
  グラファイトのまま保たれる

#### Scenario: アクセント色の切替

- **WHEN** ユーザーが設定でアクセントをゴールドとアイスブルーの間で切り替える
- **THEN** アプリ全体のアクセント点灯色が選択した 1 色へ即座に置き換わり、
  併用や混在は起きない

### Requirement: Git 状態の信号色

ファイルの Git 作業ツリー状態は、意味専用の信号色で表示しなければならない（MUST）— 新規・変更・コンフリクトをそれぞれ区別できる色で示す。変更状態の信号色は、アクセントのゴールドと視覚的に明確に区別できる色でなければならない（MUST）。ファイル種別ごとの色分けは行ってはならない（MUST NOT）。

#### Scenario: 変更のあるファイルの表示

- **WHEN** ファイルが Git 上で新規／変更／コンフリクトのいずれかの状態にある
- **THEN** その行に状態に対応する信号色のマーカーが表示され、状態のない
  ファイルにはマーカーが表示されない

#### Scenario: 信号色とアクセントの識別

- **WHEN** ゴールドのフォルダアイコンと、Git「変更」状態のマーカーが同一行に
  並ぶ
- **THEN** 両者は色相で明確に区別でき、混同されない

### Requirement: 機械加工 R と 4px グリッド

パネル・ダイアログ・メニュー等の角は 4px の角丸（機械加工 R）に統一しなければならない（MUST）。角丸の値は `PolarisTokens.radius` の単一トークンで統制し、コンポーネント側に角丸値をハードコードしてはならない（MUST NOT）。レイアウトの全寸法（高さ・余白）は 4px の倍数に乗せなければならない（MUST）。

#### Scenario: 角丸の一貫性

- **WHEN** パネル・カード・ダイアログ・メニュー等の矩形要素が描画される
- **THEN** その角は `PolarisTokens.radius`（4px）の角丸で描画され、0px の
  ナイフエッジや大きな角丸を使わない

### Requirement: アニメーションの排除

UI のフィードバックは即時（0ms）でなければならない（MUST）。ink splash・
ホバーのフェード・ページ遷移アニメーション・スクロールの慣性/バウンス/
オーバースクロールグローを使ってはならない（MUST NOT）。

#### Scenario: クリック・ホバーの反応

- **WHEN** ユーザーが行や操作対象をホバー・クリックする
- **THEN** 背景色やアクセントが次フレームで即座に切り替わり、フェードや
  リップルは発生しない

#### Scenario: スクロール

- **WHEN** ユーザーがリストをスクロールし、端に達する
- **THEN** バウンスやオーバースクロールのグロー表現は発生せず、端で停止する

### Requirement: ウィンドウのタイトルバー統合

macOS の標準タイトルバーを非表示にし、アプリ自前のトップバーをタイトルバー兼ウィンドウのドラッグ領域として機能させなければならない（MUST）。

#### Scenario: ウィンドウ表示とドラッグ

- **WHEN** アプリのウィンドウが表示される
- **THEN** macOS の標準タイトルバーは表示されず、信号機ボタンの領域が確保
  された自前トップバーが表示され、そのトップバーのドラッグでウィンドウを
  移動できる

### Requirement: タイポグラフィ方針

UI テキスト（ラベル・ファイル名・本文）はプロポーショナルフォントで描画し、桁を揃えるデータ（パーミッション・サイズ・日時・パス）は等幅フォントで描画しなければならない（MUST）。日本語テキストは適切な日本語フォントへフォールバックして正しく表示されなければならない（MUST）。

#### Scenario: データ列の桁揃え

- **WHEN** ファイル一覧でパーミッション・サイズ・日時の列が描画される
- **THEN** 等幅フォントで描画され、各行で桁が揃う

#### Scenario: 日本語ファイル名

- **WHEN** 日本語を含むファイル名・フォルダ名が描画される
- **THEN** 文字化けや欠落なく日本語フォントで表示され、列のレイアウトが崩れ
  ない

### Requirement: モノラインアイコン

フォルダ・ファイル等の型アイコンは、幾何学的なモノライン（ヘアラインのストローク）として自前描画しなければならない（MUST）。既製のアイコンフォント/アイコンセットに依存してはならない（MUST NOT）。

#### Scenario: 型アイコンの描画

- **WHEN** ファイル一覧やサイドバーでフォルダ・ファイルのアイコンが表示される
- **THEN** 自前描画の幾何学モノラインアイコンが表示される

### Requirement: 表示灯型の状態インジケーター

状態を示す小さな点状インジケーター（Git 状態・各種ステータス）は、単色の塗り点でなく、ヘアラインのリングと塗りのコアからなる同心円（「ベゼルに嵌った表示灯」）として描画しなければならない（MUST）。

#### Scenario: ステータス点の描画

- **WHEN** Git 状態やステータスを示す点状インジケーターが表示される
- **THEN** 同心円（外側リング＋内側コア）の表示灯として描画される

### Requirement: 選択モデル

ファイル一覧は複数選択に対応しなければならない（MUST）。選択された行は控えめ
な塗りで示し、主選択（アンカー）となる 1 行のみをフルアクセント色で点灯
させなければならない（MUST）。キーボードフォーカスは選択と一体で扱い、選択
とは別のフォーカス表示を設けない。

#### Scenario: 単一選択

- **WHEN** ユーザーが 1 行を選択する
- **THEN** その行が主選択としてフルアクセント色（行名・アイコン・左バー）で
  点灯する

#### Scenario: 複数選択

- **WHEN** ユーザーが複数行を選択する
- **THEN** 選択された全行が控えめな塗りで示され、主選択（最後に選んだ 1 行）
  のみがフルアクセント色で点灯する

### Requirement: 既存レイアウトの維持

Polaris の適用は視覚層のスタイル差し替えに限り、Roola の既存レイアウト — どの画面が存在するか、タブ/ペイン/サイドバーの配置、ナビゲーション構造、各画面内のウィジェットの配置順 — を変更してはならない（MUST NOT）。Polaris は視覚層の方針であって、レイアウト仕様ではない。（macOS タイトルバーの統合はウィンドウ枠の扱いであり、ここでいう既存レイアウトには含まない。）

#### Scenario: コンポーネントへの Polaris 適用

- **WHEN** あるコンポーネントに Polaris を適用する
- **THEN** 色・トーン・境界・角丸・タイポグラフィ・アイコン・インジケーター・
  操作フィードバックのみが変わり、ウィジェットツリーの構造・配置・画面遷移は
  変わらない

### Requirement: 背景の外観モード

アプリ背景の外観モードは `opaque`（不透明なグラファイト筐体）と `transparent`（筐体を半透明に合成し背後のデスクトップを透かす）の 2 種類のみを提供しなければならない（MUST）。背景を任意の単色・画像・グラデーションへ差し替えるモードを提供してはならない（MUST NOT）— Polaris のグラファイト筐体の視覚的一貫性を保つため。アプリは外観モードに依らず、UI が自前で背景を塗らない領域でも不透明なグラファイトの基底層を描画しなければならない（MUST）。

#### Scenario: 不透明モード

- **WHEN** 外観モードが `opaque` で、ターミナルや設定画面など自前で背景を塗らない
  画面が表示される
- **THEN** それらの領域も不透明なグラファイトで描画され、透過ウィンドウ素通しの
  透明な穴は生じない

#### Scenario: 透過モード

- **WHEN** 外観モードが `transparent` に設定される
- **THEN** UI ツリー全体が一括で半透明合成され、トーン階層を保ったまま均一に
  背後のデスクトップが透ける
