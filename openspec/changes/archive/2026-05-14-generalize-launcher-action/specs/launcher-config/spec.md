## MODIFIED Requirements

### Requirement: ランチャーエントリの永続化

システムは、ランチャーエントリ（一意 ID・表示名・作業ディレクトリ絶対パス・動作タイプ・アイコン画像パス）の集合をローカルストレージへ永続化し、アプリ起動時に復元 SHALL する。動作タイプは「📂 開くだけ」「⚡ コマンド実行（コマンド文字列・終了後シェル残留フラグ）」「🤖 Claude Skill（Skill 名）」のいずれかであり、永続化スキーマは sealed union として表現される。

#### Scenario: 起動時にエントリを復元する

- **WHEN** ユーザーがアプリを再起動する
- **THEN** 前回保存したランチャーエントリ一覧が（動作タイプを含めて）サイドバー / ランチャー popover に復元表示される

#### Scenario: 保存先が破損または存在しない

- **WHEN** 永続化ファイルが存在しない、または JSON として不正な状態である
- **THEN** システムは空のエントリ一覧として扱い、ユーザーへエラーを表示せず正常起動する

#### Scenario: 旧スキーマ JSON を読み込む

- **WHEN** 永続化ファイルが旧スキーマ（`repositoryPath` / `skillName` の 2 フィールド）で書かれている
- **THEN** システムは読み込み時に新スキーマへ自動変換する。`skillName` が空文字の場合は `OpenHere` 動作、非空の場合は `ClaudeSkill(skillName)` 動作として復元する。`repositoryPath` の値はそのまま `workingDirectory` として扱う

#### Scenario: 旧スキーマ JSON の書き戻し

- **WHEN** 旧スキーマで読み込まれたエントリをユーザーが編集して保存する、または新規エントリを追加する
- **THEN** システムは新スキーマ（`workingDirectory` / `action`）で永続化ファイルを書き出す。同じファイル内に旧スキーマと新スキーマが混在することは無い（既存の全エントリも合わせて新スキーマで再書き出しされる）

### Requirement: ランチャーエントリの追加

システムは、ユーザーが設定画面から新しいランチャーエントリを追加できるようにする SHALL。エントリには動作タイプを 1 つ選択し、選択タイプ固有のフィールド（コマンド文字列または Skill 名）を入力する必要がある。

#### Scenario: 「開くだけ」動作で追加する

- **WHEN** ユーザーが表示名・作業ディレクトリ・「📂 開くだけ」を選択して「保存」を押す
- **THEN** 新しいエントリが `OpenHere` 動作で永続化され、サイドバー / ランチャー popover に即時反映される

#### Scenario: 「コマンド実行」動作で追加する

- **WHEN** ユーザーが表示名・作業ディレクトリ・「⚡ コマンド実行」を選択し、コマンド文字列（例: `npm run dev`）を入力して「保存」を押す
- **THEN** 新しいエントリが `RunCommand(command, keepShellAfterExit: true)` 動作で永続化され、即時反映される

#### Scenario: 「Claude Skill」動作で追加する

- **WHEN** ユーザーが表示名・作業ディレクトリ・「🤖 Claude Skill」を選択し、Skill 名を入力（または `.claude/skills/` 候補から選択）して「保存」を押す
- **THEN** 新しいエントリが `ClaudeSkill(skillName)` 動作で永続化され、即時反映される

#### Scenario: 作業ディレクトリが存在しない

- **WHEN** ユーザーが入力した作業ディレクトリパスがローカルに存在しない
- **THEN** システムは保存を拒否し、「指定されたディレクトリが見つかりません」のバリデーションエラーを表示する

#### Scenario: 表示名が空

- **WHEN** ユーザーが表示名を未入力のまま「保存」を押す
- **THEN** システムは保存を拒否し、表示名フィールドに必須エラーを表示する

#### Scenario: 「コマンド実行」でコマンドが空

- **WHEN** 動作タイプ「⚡ コマンド実行」を選択した状態で、コマンド文字列を未入力（または空白のみ）のまま「保存」を押す
- **THEN** システムは保存を拒否し、コマンドフィールドに「コマンドを入力してください」のエラーを表示する

#### Scenario: 「Claude Skill」で Skill 名が空

- **WHEN** 動作タイプ「🤖 Claude Skill」を選択した状態で、Skill 名を未入力のまま「保存」を押す
- **THEN** システムは保存を拒否し、Skill 名フィールドに「Skill 名を入力してください」のエラーを表示する

### Requirement: ランチャーエントリの編集

システムは、既存ランチャーエントリの全フィールド（動作タイプの切替を含む）を編集できる SHALL。

#### Scenario: 既存エントリを編集して保存する

- **WHEN** ユーザーが設定画面または「管理…」メニュー経由で既存エントリの編集画面を開き、値を変更して保存する
- **THEN** 変更内容が永続化され、サイドバー / ランチャー popover の表示も更新される

#### Scenario: 動作タイプを切り替える

- **WHEN** ユーザーが既存エントリの動作タイプを別タイプに変更する（例: `ClaudeSkill` → `RunCommand`）
- **THEN** UI は新タイプ用のフィールドに切り替わり、保存時には新タイプの動作で永続化される。旧タイプ用のフィールド値（例: `skillName`）は永続化ファイルから消える

#### Scenario: 動作タイプ切替後に元タイプへ戻す

- **WHEN** ユーザーが動作タイプを切り替えた後、保存せずに元のタイプへ戻す
- **THEN** 元タイプのフィールド値はセッション内で保持されており、再表示される（編集中の値が失われない）

### Requirement: ランチャーエントリの削除

システムは、既存ランチャーエントリを削除できる SHALL。

#### Scenario: 削除確認を経て削除する

- **WHEN** ユーザーが削除ボタンを押し、確認ダイアログで「削除」を選択する
- **THEN** 当該エントリは永続化ファイルとサイドバー / ランチャー popover の両方から取り除かれる

#### Scenario: 削除を取り消す

- **WHEN** ユーザーが削除ボタンを押し、確認ダイアログで「キャンセル」を選択する
- **THEN** エントリは削除されず、何の変更も発生しない

### Requirement: アイコン画像の取り込みと保存

システムは、ユーザーが選択した画像ファイル（PNG / JPEG）を取り込み、アプリのサポートディレクトリへコピー保存し、エントリと紐付ける SHALL。

#### Scenario: アイコンを選択して登録する

- **WHEN** ユーザーがファイル選択ダイアログから画像を選び確定する
- **THEN** システムは画像をアプリのサポートディレクトリ配下にコピーし、エントリにそのパスを保存する

#### Scenario: アイコンを未指定で保存する

- **WHEN** ユーザーがアイコンを指定せずにエントリを保存する
- **THEN** システムはデフォルトアイコンを当該エントリへ割り当てる

## ADDED Requirements

### Requirement: 動作タイプの相互排他性

システムは、1 エントリに対して動作タイプを 1 つだけ持たせる SHALL。複数タイプの同時設定や、タイプ未選択の状態は永続化スキーマ上も表現できない。

#### Scenario: 永続化ファイル上の動作タイプは常に 1 種

- **WHEN** ユーザーがエントリを編集 / 新規作成する
- **THEN** 永続化される `action` フィールドは `OpenHereAction` / `RunCommandAction` / `ClaudeSkillAction` のいずれか 1 つだけを持つ（sealed union により他タイプ用のフィールドは JSON 上に出現しない）

#### Scenario: 不正な action タイプを含む JSON を読み込む

- **WHEN** 永続化ファイル中の `action.type` が新スキーマ既知値（`openHere` / `runCommand` / `claudeSkill`）以外
- **THEN** システムは当該エントリを読み飛ばし、ログに警告を出してアプリ起動は継続する（破損した 1 エントリで全体が読めなくなることは無い）

### Requirement: 「コマンド実行」動作のシェル残留オプション

システムは、`RunCommand` 動作のエントリに対し、「コマンド終了後にターミナルを残す」フラグ（`keepShellAfterExit`）を設定できる SHALL。デフォルトは true（残す）。

#### Scenario: コマンド終了後にシェルプロンプトが表示される

- **WHEN** `keepShellAfterExit=true` で登録された一発完結型コマンド（例: `make build`）を起動し、コマンドが正常終了する
- **THEN** PTY セッションは閉じられず、ユーザーのログインシェルがそのまま起動して操作可能になる

#### Scenario: コマンド終了後に PTY が閉じる

- **WHEN** `keepShellAfterExit=false` で登録されたコマンドが正常終了する
- **THEN** PTY セッションは終了し、セッション状態は「完了」に遷移する

#### Scenario: 編集画面でフラグを切り替える

- **WHEN** ユーザーが「⚡ コマンド実行」選択時のチェックボックス「コマンド終了後もターミナルを残す」を操作して保存する
- **THEN** 当該フラグが永続化され、次回起動時から新しい挙動で動作する
