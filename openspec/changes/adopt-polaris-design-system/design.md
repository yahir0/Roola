# 設計: Polaris デザインシステムの採用

> このドキュメントは、Polaris に至った設計判断とその理由を、後続セッション
> （別セッションで実装する）が経緯を再現できるよう詳細に残すことを目的とする。
> 実装の手順は `tasks.md`、要件は `specs/polaris-design-system/spec.md` を参照。

## Context

### 現状

Roola は Flutter Desktop（macOS）のターミナルランチャー。テーマは Flutter
Material 3 のデフォルトに、ADR-0020 で導入した「Win10/11 風フラット実用テーマ」
を乗せた構成（`lib/app/theme.dart` の `AppTheme` / `LogoTheme`）。角丸 2px・
elevation 0・`VisualDensity.compact`・ライト/ダーク 2 テーマ。

### この変更の発端

ユーザーから「Flutter（Material Design）のモバイル感・Android 感を完全に排除
したい」という要望。ADR-0020 の Win11 フラットも「Flutter らしさが残りすぎ」と
された。理想像のキーワードは当初、戦闘機コックピット／計器・高密度・無骨さ・
Linux のマニア感と、同時にジョナサン・アイブ的な機能美（無駄を削いだハード
ウェア感）。

### 設計プロセス（重要・この手法自体を記録に残す）

仕様を机上で固めず、**独立した単一 `main()` を持つ検討用プレビュー**
（`lib/dev/cockpit_preview.dart`）を作り、既存アプリに一切影響を与えずに
`flutter run -t` で起動して、実際の見た目をユーザーが見ながら 17 回反復した。
言葉のニュアンス（「コックピット」「アメリカ感」など）は誤解を生みやすく、
実物を見て反応する方が圧倒的に速く正確だったため。第 17 稿が Polaris の
最終形であり、実装の視覚リファレンスとなる。

この反復で「コックピット」という当初の表現は実態と合わなくなり（装飾的な
モチーフはすべて却下された）、最終的に体系は **機能主義（functionalism）**
へ着地。ディーター・ラムス → ジョナサン・アイブの系譜（「より少なく、しかし
より良く」）。体系名を **Polaris** とした（アプリ名 Roola＝移動呪文ルーラに
対する「移動のための導きの星」。思想＝不動の基準星、デザイン＝そこから引かれた
航跡、という構造を名前自体が表す）。

### 制約

- リポジトリ単体で自己完結する方針（ADR-0005）。外部の知識・ツールに依存せず、
  `docs/` と OpenSpec を読めば保守できる状態を保つ。
- 状態管理・モデル等の既存規約（Riverpod / Hooks / Freezed、`docs/architecture.md`）
  は維持。本変更は「見た目の層」の置換であり、アーキテクチャ層構成は変えない。

## Goals / Non-Goals

**Goals:**

- `lib/app/theme.dart` を Polaris のデザイントークン（`PolarisTokens`
  ThemeExtension）ベースへ全面置換する。
- `lib/ui/` 配下の全コンポーネントを Polaris トークン参照へ移行し、ハード
  コードされた色・角丸・余白を排除する。
- 機能主義に基づく一貫した視覚言語（ダーク専用・グラファイト・単一アクセント
  〔ゴールド／アイスブルー切替〕・R=4px・4px グリッド・アニメーション 0ms）を
  確立する。
- macOS タイトルバーを隠し、アプリを「1 個の機械」として見せる。
- 設計判断を ADR-0038 に記録し、ADR-0020 を Superseded にする。

**Non-Goals:**

- ライトテーマは作らない（ダーク専用と決定）。
- アプリのアーキテクチャ層（MVVM / Riverpod 構成）は変更しない。
- **Roola の既存レイアウトは一切変更しない。** Polaris はあくまで「視覚層の
  方針」であり、その適用は色・トーン・素材表現・境界・角丸・タイポグラフィ・
  アイコン・インジケーター・操作フィードバックの差し替えに限る。どの画面が
  存在するか、タブ/ペイン/サイドバーの配置、ナビゲーション構造、各画面内の
  ウィジェットの配置順は現状のまま保つ。検討用プレビューのレイアウト
  （サイドバー＋一覧＋ステータスバー）は視覚言語を見せるためのキャンバスで
  あって、レイアウト仕様ではない。（タイトルバー統合 D8 はウィンドウ枠の
  扱いであり、ここでいう既存レイアウトには含まない＝ヒアリング済みの承認
  事項。）
- 機能（ファイル操作・ターミナル・Git 等）の挙動変更はしない。見た目のみ。
- 新規パッケージ追加はしない（フォントは OS 同梱、`window_manager` は既存）。
- `cockpit_preview.dart` は製品コードではない。実装反映後に破棄する
  （恒久的に残すなら `polaris_preview.dart` 等へ改名するが、原則は破棄）。

## Decisions

各決定は「決定 / 理由 / 検討し却下した案」で記す。却下案を残すのは、後続
セッションが同じ袋小路を再探索しないため。

### D1. 思想を「機能主義」に置き、体系を Polaris と名付ける

- **決定**: デザインシステムは特定のビジュアルでなく「機能主義」という思想を
  基盤とする。思想は不変、ビジュアル（色・寸法等）はそこからの導出物で、
  機能的に優れる選択があればビジュアルは差し替えてよい。体系名は Polaris。
- **理由**: ユーザーが最も大切にしたいのは見た目そのものでなく「機能が形を
  決める」という原理。名前もそれを反映すべき、という明確な意思表示があった。
  Polaris は「移動（ルーラ）の導きの星」で、不動の原理（公理）から可変の
  デザイン（定理）が導かれる構造を一語で表す。
- **却下案**: 質感由来の名（Billet/Instrument/Gauge 等）、数学由来の名
  （Axiom/Invariant/Basis 等）。思想性・詩情・アプリ名との呼応で Polaris に。

### D2. ダーク専用にする

- **決定**: ライト/ダークの 2 テーマをやめ、ダーク専用にする。
- **理由**: 「沈んだ計器ディスプレイ」「暗い地のかっこよさ」はライトでは再現
  できず別物になる。世界観の純度を最優先。トークン構成もダーク前提で単純化
  できる。
- **却下案**: ライト併存（世界観が割れる・検証コスト倍）／将来ライト対応の
  余地だけ残す（不要な抽象化）。

### D3. 地はグラファイト。well/bg の 2 トーン階層

- **決定**: 地は純黒でなく、僅かにリフトしたグラファイト。`well`（最も暗い＝
  沈んだ計器ディスプレイ＝ファイル一覧）と `bg`（一段明るい＝筐体の枠＝
  トップバー/サイドバー/ステータスバー）の 2 トーンで「ベゼルに嵌った画面」
  というマテリアリティを出す。影は使わず、トーン差と 1px ラインのみ。
- **理由**: 一枚板の純黒は「素材のない空洞」で、初見ユーザーに怪しさ・とっつき
  にくさを与える（実機の professional な IDE と同程度の暗さが安心感の下限）。
  一方でリフトしすぎるとハッカー的な暗いかっこよさが死ぬ。純黒から僅かに
  上げるに留め、「物体感」は明るさでなく色温度・筐体上端の 1px ハイライト
  （`topEdge`）・可読なラベルで出す。well/bg の 2 トーンは「平面の図」を
  「筐体に画面が嵌った 1 個の物体」に変える最重要の構造。
- **却下案**: 純黒一枚（怪しい）／グラファイトまで明るくリフト（ハッカー感が
  死ぬ）／Nord・Gruvbox・Dracula のシンタックスパレット（くすんで見える、
  「ソフトのテーマ」に見えて機材感が出ない）。

### D4. アクセントは単一色（実装時にゴールド／アイスブルー切替へ改定）

> **改定（実装中）**: 「1 色運用」は維持しつつ、ユーザー設定でゴールド
> （デフォルト）とアイスブルー `#48C9DE` を選べる切替式とした。同時併用は
> しない（常に 1 色）。当初の「GOLD 単色固定」決定を、ICE 検討期の魅力も
> 残す形でユーザー選択へ昇格させたもの。詳細は ADR-0038 D4。


- **決定**: アクセントは `#D0A341`（削り出しの真鍮・ゴールド）。1 色のみ
  （複数アクセント禁止）。選択・現在地・フォルダ等、機能的に意味のある箇所に
  限定使用する。
- **理由**: 地（グラファイト）が寒色なので、アクセントも寒色（シアン等）だと
  全体が無機質・冷たくなる。冷たい筐体に暖色アクセント＝フェラーリのイエローが
  アルミ筐体で生きるのと同じ「生きた高級感・温度差」が出る。色は「必要な
  ところに意図を持って」使い、ファイル種別ごとの色分け（＝虚飾の虹）はしない。
- **検討経緯**: 一度はアイスシアン（`#48C9DE`）に決めかけた（意味的に中立で
  green/amber/red を信号に温存できるため）。しかし全体が寒色で無機質に
  なる点が最後まで「しっくりこない」とされ、ICE/AZURE/GOLD/VIOLET を並べて
  比較し GOLD を採用。アクセント色相は最優先で気持ちよく決めるべき、信号色は
  後から調整できる、という判断。
- **却下案**: シアン（寒色一辺倒）／複数アクセントの選択肢化（「1 色だから
  効く」の逆。なお ICE 検討期に「数色のみ選択可」という案が出たが、GOLD 単色
  確定により破棄）／フォルダだけ静穏トーン（`accentMuted`）にする案も実装して
  試したが「さっきの方が好み」で却下、フォルダはフルゴールドに戻した。

### D5. 信号色 green/amber/red を Git 状態で使う

- **決定**: green=新規 / amber=変更 / red=コンフリクト を、ファイル行左の
  状態マーカーと、ステータスバーの Git サマリで使う。意味専用。
- **理由**: 「増やしていい色は意味を運ぶ色だけ」。Roola は Git ビューを持ち
  （ADR-0030）、エクスプローラ自体が git リポジトリを映すので、Git 状態色は
  実際に機能する多色＝正当。変化のある行にしか出ないので信号として効く。
- **要対応**: アクセントがゴールド（暖色）になったため、git「変更」の amber と
  色相が近い。**git「変更」の信号色を amber から、ゴールドと十分に区別できる
  別トーンへ振り直す**こと（spec の要件参照）。
- **却下案**: ファイル種別ごとの色分け（全部が色付くと色が信号でなくなる）。

### D6. 角は機械加工 R=4px、全寸法は 4px グリッド

> **改定（実装中）**: 当初 R=2px としたが、設定画面など大きめのパネルで
> 2px が実質直角に読めて硬すぎたため R=4px へ引き上げた。角丸は
> `PolarisTokens.radius` の単一トークンで統制し、`BorderRadius.circular(2)`
> のハードコードは全廃済み。詳細は ADR-0038 D6。

- **決定**: 角丸は 0px でなく 4px（`PolarisTokens.radius` トークン）。全寸法
  （行高・バー高・余白）を 4px の倍数に乗せる。ファイル一覧は 8px の
  インセットを取り、R=4px・1px ボーダーの「計器ディスプレイパネル」として
  ベゼルに嵌め込む。
- **理由**: 完全な 0px のナイフエッジは安っぽく不安定。削り出し部品のような
  ごく僅かな面取りが工業製品感を出す（ADR-0020 が 2px を選んだ理由とも整合）。
  4px グリッドの厳密さ＝精密さそのものが機械の質感になる（不揃いな余白が
  無意識の「惜しさ」を生む）。
- **却下案**: 0px ナイフエッジ（安っぽい）／ADR-0020 の 2px をそのまま流用
  （角の値は同じだが、Polaris では「計器ディスプレイパネルとしてインセットして
  嵌め込む」ことで初めて R が意味を持つ）。

### D7. アニメーション 0ms（即時フィードバック）

- **決定**: ink splash・ホバーのフェード・ページ遷移アニメ・スクロールの
  バウンス/グローをすべて排除。状態変化は次フレームで即座に。
- **理由**: 機能主義＝即応。スイッチを押したような即時フィードバックが計器・
  道具らしさ。フェードやバウンスは「Flutter らしい柔らかさ＝モバイル感」の
  発生源。
- **実装上の要点**: `splashFactory: NoSplash`、ボタンの `animationDuration:
  Duration.zero`、ホバーは Material の state machine を経由せず `MouseRegion`
  自前管理、`PageTransitionsBuilder` で遷移を即時化、`ScrollBehavior` で
  オーバースクロール表現と慣性を排除（`ClampingScrollPhysics`）。

### D8. macOS タイトルバーを隠し、トップバーに統合

- **決定**: `window_manager` の `TitleBarStyle.hidden` でタイトルバーを隠し、
  自前のトップバー（アプリ名の行）をタイトルバー兼ドラッグ領域（`DragToMoveArea`）
  にする。左端は信号機ボタンの領域（約 78px）を確保。
- **理由**: アプリ全体を「1 個の機械」に見せる。OS のバーが暗い筐体の上に
  乗ると物体としての一体感が崩れる。
- **却下案**: 標準タイトルバー（実装は素直だが世界観が弱い）／枠なし完全自作
  （ウィンドウ操作を全部自作する必要があり過剰）。

### D9. タイポグラフィ: SF Pro（UI）+ SF Mono（データ列）

- **決定**: クローム（ラベル・ファイル名・本文）は OS 標準のプロポーショナル
  （SF Pro）。データ列（perm / size / 日時 / パス）は等幅（SF Mono、フォール
  バック Menlo）。日本語は Hiragino Sans へフォールバック。本文ウェイトは
  **w600** を基準、全大文字ラベルは w600、ロゴワードマークのみ w700。
- **理由**: 全面等幅は「ターミナル/コードエディタの安っぽい字面」になり却下。
  桁を揃えたいデータだけ等幅にすれば、ターミナル感を出さずに整列が得られる。
  Flutter macOS は native（AppKit）のような stem darkening をせず暗い地で
  文字が痩せて見えるため、ウェイトを w600 に上げて字面に肉を持たせる。
- **却下案**: 全面等幅（安っぽい・エディタに見える）／Futura 等の特徴的書体
  （古臭く見えた）／w400・w500（暗い地で痩せる）／w700（太すぎた）。
- **補足**: 日本語を等幅グリッドの中で桁揃えする必要が将来生じた場合は、
  Sarasa Term J（ADR-0017 で同梱済み・CJK 字幅が ASCII の 2 倍）を使う。
  現状のデータ列は ASCII のみなので不要。

### D10. アイコンは幾何学モノラインの自作描画

- **決定**: フォルダ／ファイル／山括弧などのアイコンは `CustomPaint` で
  幾何学モノライン（ヘアラインのストローク）を自作描画する。Cupertino /
  Material の既製アイコンは使わない。
- **理由**: 既製アイコンは consumer 向けで丸み・親しみが強く、精密な計器 UI の
  中で温度が合わない。

### D11. 状態インジケーターは「表示灯」型

- **決定**: ステータス点（Git 状態・電源灯・Ready 表示等）は単色の塗り点でなく、
  同色のヘアラインのリング（淡）＋塗りのコア（濃）の同心円＝「ベゼルに嵌った
  表示灯」として `CustomPaint` で描く。
- **理由**: 単色のベタ点は安っぽい。実機の計器・機材の表示灯の構造を写すと、
  小さな点にも精密さ・素材感（高級感・工業製品感）が出る。

### D12. 選択モデル: 主選択のみフルアクセント点灯

- **決定**: ファイル一覧の選択行は控えめな塗り（`surfaceHi`）。複数選択時、
  主選択（アンカー＝最後に選んだ 1 つ）だけがフルアクセント色で点灯
  （行名・アイコン・左 2px バー）。キーボードフォーカスは選択と一体化し、
  別表示しない。
- **理由**: 全選択行をフルアクセントにすると画面がアクセントだらけになり信号性
  が薄れる。「今アクティブな 1 つ」だけを点灯させると計器的で、機能的に意味の
  ある目立ち方になる。フォーカスと選択を分けるとマウス選択とキー移動位置が
  混ざるので一体化する。
- **却下案**: 全選択行フルアクセント（アクセント過多）／フォーカスを別の
  1px 枠で表示（複雑・本ツールでは一体化で十分）。

### D13. 装飾モチーフは一切足さない

- **決定**: 「機械っぽさ」を出すための applied な装飾（四隅のレジストレーション
  マーク、選択の照準ブラケット、特徴的書体の引用等）は使わない。機械感は
  構造・精密さ・抑制（トーン階層・グリッド・1 アクセント）からのみ滲ませる。
- **理由**: 反復の途中でこれらを試したが、すべて「装飾過多・古臭い」として
  却下された。機能美＝機能が形を決める、であって、機械風モチーフの追加では
  ない。この原則自体が Polaris の核。

### D14. 背景の外観機能を Polaris と整合させ、透過 / 不透明の 2 択に絞る

> **追加（実装後の詰め / tasks 7.6）**: Polaris 移行で全サーフェスが不透明
> グラファイトを塗るようになり、背面の外観レイヤーが隠れて 透過・単色 が
> 機能停止していた。その修正にあたり、外観機能と Polaris の両立を見直した。

- **決定**: 外観モードを `opaque`（不透明なグラファイト筐体）と `transparent`
  （筐体ごと半透明にし背後のデスクトップを透かす）の 2 択に絞り、`solid` /
  `image` / `gradient` を廃止する。旧設定値は `opaque` へフォールバック移行
  する。透過は単一の `Opacity` で UI ツリー全体を一括合成して実現し、
  `_AppearanceLayer` は両モードとも不透明グラファイトの基底層を必ず敷く。
- **理由**: Polaris はダーク専用（D2）・グラファイト筐体（D3）を視覚的
  アイデンティティとするため、背景を任意の単色・画像へ差し替えるモードは
  思想と相反する。透過は「グラファイトが薄くなって背後が透ける」だけで世界観を
  壊さず、背後のデスクトップを見るという機能的価値もあるため残す。面ごとに
  半透明化するとトーン階層（`well` / `bg`）が重なる箇所でアルファが二重掛けで
  濁るため、`Opacity` でツリー全体を一括合成する。基底層は、自前で背景を塗らない
  画面（ターミナルのネイティブビュー周り・設定画面）が透過ウィンドウ素通しの
  透明な穴になるのを防ぐ。
- **検討し却下した案**: 全モード（透過/単色/画像/グラデ）維持（グラファイト
  筐体と相反）／外観機能の全廃（透過は世界観を壊さず機能的価値があり惜しい）／
  面ごとに alpha を持たせる（トーン階層が重なる箇所で濁る）。

## Risks / Trade-offs

- **[全コンポーネント横断の大規模変更]** → `PolarisTokens` を先に確立し、
  コンポーネントは段階移行（`tasks.md` でファイル単位にタスク分割）。各段階で
  `flutter analyze` を通す。
- **[ダーク専用にしたことで、明所での視認性・好みに対応できない]** → 機能主義の
  純度を優先する明示的な決定（D2）。将来要望が出れば別 change で扱う。
- **[ゴールドアクセントと git「変更」amber の色相衝突]** → git「変更」信号色を
  振り直す（D5）。spec に要件として明記。
- **[`cockpit_preview.dart` が `lib/` 配下に残ると `flutter analyze` の対象に
  なり、放置すると陳腐化する]** → 実装反映後に破棄（D2 Non-Goals）。実装中の
  視覚リファレンスとしてのみ使う。
- **[Flutter macOS のテキスト描画は native と差があり、デザイン値が実機で
  痩せ/滲んで見えうる]** → ウェイト w600 基準で吸収（D9）。`textScaler` を
  1.0 に固定する余地も残す。
- **[ハードコードされた色・角丸・余白の見落とし]** → 移行時に `Color(0x` /
  `BorderRadius` / マジックナンバーの余白を grep で洗い出す。

## Migration Plan

1. ADR-0038 を追加（Polaris の思想・トークン構成・各決定を記録）。
   ADR-0020 を `Status: Superseded by ADR-0038` にする。
2. `lib/app/theme.dart` を `PolarisTokens`（ThemeExtension）+ ダーク専用
   `ThemeData` ビルダーへ全面置換。`LogoTheme` は廃止または Polaris トークンへ
   吸収。
3. `lib/ui/` 配下を領域ごと（explorer → workspace → git → settings → common
   → app）に Polaris トークン参照へ移行。各領域完了ごとに `flutter analyze`。
4. macOS タイトルバー統合（`window_manager` 設定の拡張）。
5. `docs/architecture.md` / `CLAUDE.md` のテーマ関連記述を更新。
6. `lib/dev/cockpit_preview.dart` を破棄。
7. ロールバック戦略: 本 change は単一の feature ブランチで進め、問題時は
   ブランチごと破棄すれば現行テーマに戻る（`theme.dart` 置換が中心のため）。

## Open Questions

  あるいは別色相）。実装時に Polaris のパレット内で確定する。
- `cockpit_preview.dart` を破棄するか、`polaris_preview.dart` として当面残すか
  （原則は破棄。残す価値があれば別途判断）。
- 第 17 稿プレビューにある検証用トグル（密度 DENSE/ULTRA、PERM 列表示）を

  するため、実装時に既存仕様と突き合わせる。
