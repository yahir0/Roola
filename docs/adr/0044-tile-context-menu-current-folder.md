# ADR-0044: タイル右クリックメニューに「現在のフォルダ」項目を追加する

- **Status**: Accepted
- **Date**: 2026-05-21

## Context

エクスプローラの右クリックメニューは現状 3 経路ある（ADR-0021 / ADR-0026 ベース）。

| 右クリック対象 | 表示されるメニュー | 操作対象 |
| --- | --- | --- |
| ファイルタイル | `showFileContextMenu` | クリックしたファイル |
| ディレクトリタイル | `showExplorerContextMenu(node)` | クリックしたディレクトリ |
| リスト下部の余白（`_CurrentDirBackdrop`） | `showExplorerContextMenu(currentDir, showRename=false, ...)` | 現在表示中のフォルダ |

「現在表示中のフォルダ」に対する操作（Open Terminal Here / Skill 実行 /
お気に入り追加 / New Folder / Paste 等）は、3 番目の余白でしか到達できない。
ところがファイルタイルはビューポートの全幅 × 行高 28px（ADR-0024 /
ADR-0038 D3）で詰めて並んでおり、ファイル数がビューポートを埋めるとリスト
内に右クリックできる「隙間」が無くなる。結果、ユーザーは下端までスクロー
ルして 160px の backdrop に到達しないと「現在のフォルダ」操作にたどり着け
ない。

## Decision

**ファイル / ディレクトリのタイル右クリックメニュー末尾に「`<folder-icon>`
`<name>` で操作 ▸」項目を追加する**。選択すると、`showExplorerContextMenu`
を **カレントフォルダを node として再度開く**。

```
[file_a.txt] を右クリック
├ 開く / Reveal in Finder / …
├ 名前変更 / コピー
├ ゴミ箱へ / プロパティ
└ 📂 myrepo で操作 ▸           ← クリックで↓を開く
                 │
                 ▼
   ├ ← 戻る                    ← クリックで↑に戻る
   ├ ─────────────
   ├ ターミナルを開く
   ├ Claude を開く
   ├ お気に入りに追加
   ├ 新規フォルダ / 新規ファイル / 貼り付け
   ├ パスをコピー / プロパティ
   └ Skill: tdd を即実行 / ランチャー登録
```

### 「戻る」の導線

drill-down で 2 段目（カレントフォルダ用メニュー）に移ると元の 1 段目は
閉じている。誤って開いた / 元のタイル操作に戻りたい時の救済として、
**2 段目の先頭に「← 戻る」項目を出す**。選択すると 2 段目が閉じ、元の
タイルメニュー（file or directory）を **同じ位置・同じパラメータで** 再オー
プンする。

実装は `showExplorerContextMenu` に `Future<void> Function()? onBack` を
持たせ、`null` でないときに先頭の「戻る」項目を出す。`_showCurrentFolderMenu`
を呼ぶ側（ファイル / ディレクトリのタイルメニュー）がそれぞれ自分自身を
再オープンする closure を `onBack` として渡す。`onBack` は再帰的にも引き
継がれるので「3 段目から 2 段目に戻り、さらに 1 段目に戻る」も可能。

### ラベルの言い回し

「現在のフォルダ (`<name>`)…」のような状態を表す名詞句ではなく、
**「`<name>` で操作 ▸」のようにアクション指向で末尾に「▸」を付けた**。理由：

- 名詞句だと「クリックしてこのフォルダに移動するのかな？」と誤解させる
- 「…」は OS によって意味が違う（macOS: ダイアログが開く / Windows: ）。
  「▸」のほうがサブメニューを示すアイコンとして直感的
- 「で操作」と書くことで、項目クリック後にメニューが続くことを示唆する

UI 上は「サブメニュー」だが、実装上は `showMenu` のネストではなく
**1 段目を閉じてから 2 段目を同じ位置で開き直す drill-down**。

### 二重表示の抑止

`showExplorerContextMenu` に `bool showCurrentFolderSection = true` を足
し、`node.path == currentPath` のときと、再オープン時（`false` を渡す）
に「現在のフォルダ」項目を出さない。`_CurrentDirBackdrop` 経由の呼び出し
では `node = currentDir` なので、追加の引数指定をしなくても自動的に抑止
される。

### 「現在のフォルダ」操作のサブセット

`_CurrentDirBackdrop` と同じく `showRename=false / showCopy=false /
showDelete=false` で開く。カレントフォルダ自身のリネーム・コピー・削除は
同画面でやらせない（メンタルモデル的に「今居る場所」を消す操作は明示的に
1 つ上の階層から行う方が安全）。

## Why

- **レイアウトを変えない**: 行間 gap を入れる / 右ガッタを設ける案は、
  Polaris の「走査線のように密に並べる」（ADR-0038 D3）と矛盾する。タイル
  右クリックに項目を 1 行足すだけならデザイン原則を壊さない
- **常に 1 右クリックで到達**: どのタイルを右クリックしても末尾に出るので、
  「下端までスクロール」「狭い隙間を狙う」のどちらも要らない
- **drill-down で十分**: Flutter の `showMenu` はネイティブのホバー式サブ
  メニューを直接サポートしない。`MenuAnchor` への移行は影響範囲が大きい
  わりに、エクスプローラの右クリックは「中身を読んで 1 つ選んで終わる」
  操作なので、2 段目を開き直す体験で実用上問題ない
- **既存コードの差分が小さい**: `showExplorerContextMenu` に 1 引数追加、
  `_handleDirectoryAction` に 1 ケース追加、`_FileAction` に 1 値追加。
  既存のメニュー組み立てロジック・ハンドラはそのまま流用

## 代替案

### 代替案 1: 行間に 2–4px の gap を入れ、gap 上の右クリックでカレント
フォルダメニューを出す

- ファイルリストが Finder ライクに「ファイルとファイルの隙間」で右クリック
  できるようになる
- Polaris D3（走査線のように密に詰める）と矛盾。密度設定（comfortable /
  compact）の意味付けも揺らぐ
- 却下。

### 代替案 2: パスバーの横に「現在のフォルダのメニューを開く」アイコンボタンを置く

- 1 クリックで届くが、「右クリックメニューがそこにある」という直感性は
  落ちる。新しいトグル UI が増える
- ボタンの存在を見つけるまでに学習が要る
- 却下。

### 代替案 3: メニューの末尾にサブメニューではなくフラットに項目を全部入れる

- 「現在のフォルダ」セクションをそのまま 8〜10 項目展開してファイルタイル
  メニューに連結する
- 1 メニュー当たり 18〜20 行に膨らみ、Polaris の「コンテキストメニューは
  視認 2〜3 秒で終わる粒度」（ADR-0038 周辺の運用）から外れる
- 却下。

### 代替案 4: `MenuAnchor` + `SubmenuButton` でホバー式サブメニューにする

- ネイティブな見た目になるが、`showMenu` 経路（`showFileContextMenu` /
  `showExplorerContextMenu`）を `MenuAnchor` に書き換える必要があり、
  メニュー項目を全部リファクタすることになる
- 「現在のフォルダ」だけ別経路で出すのも一貫性を欠く
- 体験のメリットに対して工数が見合わない。将来 `MenuAnchor` 化を一括で行う
  ときに合わせて移行する
- 当面は却下。

## Trade-offs

- **2 段目の開き直しに 1 フレーム程度の間が空く**: ネイティブのサブメニュー
  と違って、1 段目が一度閉じてから 2 段目が同じ位置に開く。視覚的にチラつ
  きが見えるケースがある。実用上は許容
- **メニューの末尾に「自フォルダへのリンク」が並ぶことの違和感**: macOS
  Finder のメニューにはこの項目は無い。Roola 固有の操作（Skill 実行・
  ターミナル起動）にカレントフォルダから即アクセスしたい、というツールと
  しての性格を優先
