# ADR-0028: ワークスペースレイアウトの永続化とターミナル再 spawn

- **Status**: Accepted
- **Date**: 2026-05-15

## Context

ADR-0026 で導入する 3 ペイン × タブのワークスペースは、再起動をまたいで復元できる
必要がある。一方で、

- ターミナルタブの PTY セッションはプロセスであり、プロセス跨ぎでは復元できない
  （出力履歴・プロセス状態は失われる）
- マルチウィンドウは別プロセス起動（ADR-0012）で、複数プロセスが同じ永続ファイルを
  書き得る

「何をどこまで永続化するか」を決める必要がある。

## Decision

### 保存先は専用 `workspace.json`

`WorkspaceLayout`（3 スロット・タブ・アクティブタブ・スプリッタ比率）を、
`ExplorerSettings`（`repo_explorer_settings.json`）とは分離した新規 `workspace.json`
に保存する。`AppPaths` に保存先を追加し、DTO（`json_serializable`）+ Repository +
`AsyncNotifier` を立てる。

保存トリガ: タブの開閉 / アクティブタブ変更 / エクスプローラの `navigateTo` /
スプリッタ比率変更。

### エクスプローラはパス復元、ターミナルは再 spawn

- **エクスプローラタブ**: カレントパスを保存し、起動時はそのパスで復元する
- **ターミナルタブ**: 作業ディレクトリ + `LauncherAction`（`type` 含む）+
  `displayName` を保存し、起動時は同じ引数で PTY を**再 spawn** する。出力履歴は
  引き継がない

`workspace.json` が無い / JSON として不正 / 全スロットが空 → 既定 3 ペインを seed
する（ADR-0026）。

### マルチウィンドウは last-write-wins

複数ウィンドウ（別プロセス）が `workspace.json` を書く競合は **後勝ち** で許容する。
各ウィンドウは起動時に 1 回読み、以後は自プロセスのレイアウトを書き戻すだけにする。
ロック・マージ・per-window ファイル分割は行わない。

## Why

- **保存先を分離した**: ワークスペースレイアウトはエクスプローラ設定（root / お気に
  入り / 表示密度）とは更新頻度も粒度も違う。1 ファイルに混ぜると `ExplorerSettings`
  のスキーマが肥大し、保存トリガも絡む。責務ごとにファイルを分ける
- **PTY は再 spawn が現実解**: PTY セッションのプロセス跨ぎ復元（出力履歴の
  シリアライズ・プロセス再アタッチ）は実装コストに見合わない。既定の素のシェルは
  再 spawn すれば実用上同じ。`RunCommandAction` の one-shot 系は再実行になるが、
  これは受け入れ仕様とする
- **last-write-wins で十分**: マルチウィンドウは ADR-0012 で「当面シンプルに」と
  決めている。ワークスペースの競合解決に複雑な調停を持ち込むのは過剰。ファイルが
  壊れても起動時の不正検知で既定 seed に倒れるため、最悪でもクラッシュしない

## 代替案

### 代替案 1: PTY セッションを永続化して復元する

出力履歴をシリアライズし、再起動後に擬似的に復元、またはプロセスを再アタッチする。

- 出力履歴のシリアライズ容量、ANSI 状態の再現、プロセス再アタッチの可否など
  難点が多い
- 再 spawn で実用要求は満たせる
- 却下。

### 代替案 2: `ExplorerSettings` に `workspace` キーを足して同居させる

新ファイルを増やさず既存設定ファイルに相乗りする。

- スキーマ肥大と保存トリガの絡みを招く。表示密度の保存とレイアウトの保存が
  同じ `save()` を共有するのは責務として不適切
- 却下。

### 代替案 3: マルチウィンドウ向けに per-window 永続ファイルを分ける

ウィンドウ ID ごとに `workspace-<id>.json` を持つ。

- ウィンドウ ID の安定的な採番・GC が必要で、別プロセス起動方式（ADR-0012）と
  相性が悪い
- 「最後に閉じたウィンドウの状態で次回開く」程度の挙動で要求は足り、last-write-wins
  で十分
- 却下（マルチウィンドウの仕様を厚くするなら別 ADR）。

## Trade-offs

- **ターミナルの出力履歴は再起動で消える**: ユーザーが「履歴が消えた」と感じ得る。
  既定の素のシェルでは影響が小さい。仕様として明示する
- **マルチウィンドウで他ウィンドウのレイアウトを上書きし得る**: 複数ウィンドウを
  使う運用では「最後に保存したウィンドウ」が勝つ。意図的な仕様として受容する
- **保存頻度**: `navigateTo` ごとに保存するため書き込み回数が増える。JSON 1 ファイル
  かつサイズが小さいため実害は無いが、必要ならデバウンスを後追いで足せる
