# ADR-0033: コマンドレジストリとネイティブメニューバーによる統一ショートカット機構

- **Status**: Accepted
- **Date**: 2026-05-18

## Context

Roola は開発者向けの汎用ターミナルランチャーだが、キーボードショートカットは「パスのコピー（修飾キーなし `C` 2 連打、ADR-0021）」の 1 つしかない。エクスプローラのファイル操作・タブ操作・ナビゲーション・Git 操作はすべてマウス（クリック / 右クリックメニュー）専用で、キーボード操作性が不足している。

ユーザーは「全アクションにショートカットを割り当てる」「設定画面で確認・編集できる」「右クリックメニューに割り当てキーを表示する」「ユーザーが任意に変更できる」ことを求めた。あわせて `C` `C` 連打の廃止と、macOS ネイティブメニューバーの追加が合意されている。

ここで設計上の制約が 2 つある:

1. **ターミナルのフォーカス問題**: ADR-0031 によりターミナルは SwiftTerm のネイティブ `AppKitView` で描画される。ターミナルにフォーカスがあるとき、キーイベントはネイティブビューが消費し、Flutter のフォーカスツリー（`Shortcuts`）には届かない。開発者向けツールとして、ターミナル作業中もアプリのショートカットが効くことは必須。
2. **コマンドの三重定義リスク**: ショートカット・コンテキストメニュー・設定画面の 3 つの UI が、それぞれ別々にアクションを定義すると、カスタムキーの反映漏れや表示の不整合が起きる。

## Decision

アプリの全アクションを安定 ID（`CommandId` enum）で一元管理する **コマンドレジストリ** を設け、ショートカットの発火を **macOS ネイティブメニューバー（`PlatformMenuBar`）に一本化** する。

- **`CommandRegistry`**: `CommandId → CommandMetadata`（カテゴリ・日本語ラベル・アイコン・既定キーコンビ）の静的定義。全コマンド定義の単一の真実。
- **キーバインド永続化**: ユーザーのカスタム割り当てを `<appSupport>/keybindings.json` に保存する（`appearance` と同じ DTO ⇄ モデル分離 + Repository + `AsyncNotifier`）。`effectiveKeybindingsProvider` が既定 + 上書きをマージする。
- **`PlatformMenuBar` 一本化**: ショートカットは `PlatformMenuBar` の key equivalent で発火する。Flutter の `Shortcuts`/`Actions`/`Intent` は使わない。
- **`CommandDispatcher`**: `CommandId` を受け取り、フォーカス中タブ / 選択アイテムを provider から解決して実処理へ委譲する。メニューバー・コンテキストメニューの両方がここを通る。
- **設定画面**: 専用ページ `KeybindingsPage` で全コマンドをカテゴリ別に一覧・編集する。キー重複は検出して警告し、保存をブロックする。
- **`C` `C` 連打の廃止**: `copyPath` コマンドを統一システムへ移行する（既定 ⌘⇧C）。

本 ADR は ADR-0021 の `C` `C` 連打部分を supersede する（ダブルクリック式の操作モデルは維持）。

## Why

### なぜネイティブメニューバー一本化か（Flutter `Shortcuts` を使わない）

macOS のネイティブメニューバーの key equivalent は、ファーストレスポンダ（どのビューにフォーカスがあるか）に関係なく発火する。ADR-0031 でもこれを SwiftTerm 採用の利点として挙げており、ターミナルにフォーカスがあってもアプリのショートカットが効く。

Flutter の `Shortcuts` はフォーカスツリーに依存するため、SwiftTerm のネイティブビューにフォーカスがあるとキーを受け取れない。`Shortcuts` とメニューバーを併用すると、同一コマンドが 2 系統で定義され、カスタムキーの反映漏れ・不整合のリスクが生じる。単一機構（メニューバー）に統一することで、ターミナルフォーカス問題を解消しつつ定義の一元性を保つ。

### なぜコマンドレジストリ（`CommandId`）か

ショートカット・コンテキストメニュー・設定画面の 3 UI が同じ `CommandId` を介して同じメタデータ（ラベル・アイコン・キーコンビ）とディスパッチを共有する。これにより「右クリックメニューにショートカットを表示」「設定画面で一覧・編集」「カスタムキーを全 UI に即反映」が、定義を 1 箇所に置いたまま自然に実現する。

### `C` `C` を廃止する理由

ADR-0021 は Cmd+C との意味的衝突を避けるため、また Vim の `yy` 慣習に倣って `C` 2 連打を採用した。しかし全アクションを統一的なカスタマイズ可能ショートカットにするにあたり、連打式は次の点で統一システムに馴染まない:

- 連打シーケンスは `MenuSerializableShortcut` で表現できず、メニューバーに載せられない。
- 設定画面のキーレコーダ（「修飾キー + キー」を 1 回キャプチャ）と編集 UI が連打式と二系統になる。
- コンテキストメニューへの `(C C)` 表示は可能だが、他コマンドの `(⌘⇧C)` 形式と不揃いになる。

`copyPath` を「修飾キー + キー」1 コンビへ移行し、全コマンドを 1 つの形式・1 つの編集 UI・1 つの発火機構に揃える。

## 代替案

### 代替案 1: Flutter `Shortcuts`/`Actions` を主機構にする

Flutter 標準の宣言的なショートカット機構。`Intent` を介してコマンドを再利用でき、設定画面との親和性も高い。しかしフォーカスツリー依存のため、SwiftTerm のネイティブビュー（ターミナル）にフォーカスがあるとキーを受け取れない。開発者向けツールでターミナル作業中にショートカットが効かないのは要件を満たさない。**却下。**

### 代替案 2: `Shortcuts` とメニューバーの併用

メニューバーに載らない（または載せたくない）コマンドを `Shortcuts` で、残りをメニューバーで扱う案。発火経路が 2 系統になり、カスタムキーをどちらにも反映する二重管理が発生する。ターミナルフォーカス時に効くか効かないかがコマンドごとに変わり、ユーザーから見て予測不能。**却下。** 全コマンドをメニューバーに載せて一本化する。

### 代替案 3: 設定画面のセクション内インライン編集

専用ページを設けず、既存の設定画面の「キーボードショートカット」セクション内に編集 UI を詰める案。コマンド数が 30 前後あり、他セクションと同一スクロール上に並べると見通しが悪い。**却下。** 専用ページ `KeybindingsPage` とする。

## Trade-offs

- **メニュー key equivalent はテキストフィールド編集中も発火する** — macOS の仕様。既定キーから ⌘C / ⌘V / ⌘X / ⌘A / ⌘Z を除外し、標準的なテキスト編集との競合を避ける。ユーザーがあえてこれらを割り当てた場合の OS 標準キーとの競合は検出しない（衝突検出はアプリ内コマンド間のみ）。
- **修飾なし単キーは割り当て不可** — メニュー key equivalent が常時発火するため、修飾なし単キーを許すとタイピングと衝突する。レコーダで修飾キー必須をバリデーションする。`C` `C` 廃止と整合する。
- **コンテキスト依存コマンドのメニュー項目は v1 では常時 enabled** — 選択アイテムやフォーカス中タブが無いときの動的な無効化（グレーアウト）は実装しない。対象が解決できないコマンドは no-op とする。動的 enabled 制御は将来課題。
- **`C` `C` に慣れたユーザーの再学習** — ADR-0021 で導入した `C` `C` を廃止する。設定画面で `copyPath` の既定キー（⌘⇧C）を表示し、ユーザーは任意のキーへ変更できる。
- **`CommandId` enum の増減** — 永続化 DTO は `CommandId` を `name` 文字列でシリアライズし、読み込み時に未知の値を読み飛ばす。enum の追加・削除でユーザーの `keybindings.json` が壊れない。
