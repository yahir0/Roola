# ADR-0019: ランチャーをフォルダで 1 階層グループ化する

- **Status**: Accepted
- **Date**: 2026-05-14

## Context

Roola はユーザーがディレクトリ + 動作のセットを登録して使う汎用ターミナルランチャー（ADR-0016）。
登録エントリが増えてくるとサイドバーがフラットリストでは把握しづらくなる。Chrome のお気に入りや macOS
Finder のサイドバーグループのように、関連エントリをフォルダにまとめたいというフィードバックが入った。

## Decision

**ランチャーエントリを 1 階層のフォルダでグルーピングできるようにする。**

- `LauncherFolder { id, name, createdAt }` を新規追加。フォルダの中にフォルダは作らない（フラット 1 階層）
- `LauncherEntry` に `folderId: String?` フィールドを追加。null は root（未分類）
- 永続化スキーマは `{"folders": [...], "entries": [...]}` に拡張
- サイドバーは inline expand/collapse 形式でフォルダを表示。caret + フォルダアイコン + 名前
- 管理画面は常時展開のグループ化リスト。フォルダ追加 / リネーム / 削除を AppBar / popup menu で提供
- エントリ編集ダイアログに「フォルダ」ドロップダウン（フォルダなし + 既存一覧）

## Why

### ネスト深さを 1 階層に絞った理由

ランチャー用途では数十件オーダーが上限と想定（リポジトリ + よく使うコマンドの組合せ）。Chrome の
ブックマーク（数百〜数千件）と違って、2 階層以上必要になるケースが想像しづらい。一方で多階層を許すと:

- サイドバーで深さに応じた可視化が必要（インデント・親フォルダ表示）
- 移動先選択 UI が tree picker になり実装重い
- 循環防止のバリデーション
- 永続化スキーマの破壊的変更リスクが大きい

1 階層に絞ることで、フォルダの実装が「フォルダ配列 + エントリの folderId ポインタ」のフラットな
データ構造に収まり、UI も「フォルダ → エントリ」の 2 段で済む。将来必要になったら ADR を更新して
多階層化する余地はある。

### 永続化スキーマを `{folders, entries}` に拡張した理由

旧スキーマは `{entries: [...]}` のみ。folders を追加するには:

**選択肢 A**（採用）: 同じファイルに folders キーを追加 → `{folders, entries}` 形式
**選択肢 B**: フォルダ用に別ファイル（`launcher_folders.json`）を作る

A を選んだ理由:

- atomic な変更がしやすい（フォルダ削除 → 配下エントリの folderId クリアを 1 ファイル書き換えで完結）
- 1 つのスナップショットとして一貫性が保てる（A だと両方が同期した状態しか保存できない）
- 旧スキーマからの lazy migration が簡単（folders キーが無ければ空配列扱い）

### Repository 設計

`LauncherEntryRepository` と `LauncherFolderRepository` の 2 つに分けたが、内部実装は共通の
`LauncherCatalogStore` を介して同一ファイルを読み書きする。各 repository は「自分の領分のキー」だけ
を更新し、他方の配列は store から読んだ値をそのまま書き戻すため、相互干渉しない。

フォルダ削除時には配下エントリの folderId を null にクリアする副作用がある。これは
`LauncherFolderRepositoryImpl.delete` 内で store の load → modify both arrays → save で完結させる。

### サイドバー UI: inline expand/collapse

Chrome のブックマークバーのようなポップアップ型は、垂直サイドバーの上下スクロールと相性が悪く、
folder 中身を見るたびに clear/open が必要で疲れる。Finder サイドバー / VS Code のエクスプローラと
同じ caret トグル方式が macOS ユーザーに馴染みやすい。expanded 状態はセッション内で
`useState<Set<String>>` で保持し、永続化はしない。永続化用のキー / マイグレーションコストに見合うほどの
価値が無いため。デフォルトは全フォルダ展開で、新規フォルダ作成時にも展開状態として扱う。

### 管理画面 UI: 常時展開のグルーピング

管理画面は編集 UI なので、サイドバーと違って折り畳まない。フォルダごとに薄い背景色のセクション
ヘッダ + popup menu（リネーム / 削除）+ 中身のエントリ、最後に root の「未分類」セクションを表示する。

### エントリ編集: フォルダ選択は dropdown

DropdownButtonFormField で `[フォルダなし, ...既存フォルダ]` から 1 つ選ぶ。フォルダ自体の新規作成は
管理画面の「+ フォルダ」ボタン経由に統一して、編集フォーム内で folder の追加と entry の編集が混在
しないようにする。

## Trade-offs

- **多階層の拡張は破壊変更**: 将来 folder に parentFolderId を追加するには、永続化スキーマと
  既存の `folderId` の意味を再定義する必要がある。1 階層で要件が満たせている間は問題ない。
- **expanded 状態の永続化なし**: ウィンドウを閉じると展開状態がリセット。気になる場合は
  別 ADR で `explorerSettings` に追加する。
- **空フォルダの存在**: フォルダだけ作ってエントリを入れない状態を許容する。空フォルダは管理画面で
  「（このフォルダは空です）」のヒントを出す。サイドバーでは expand しても何も出ないだけ。
- **管理画面のドラッグ&ドロップ**: エントリは `LongPressDraggable` で長押しドラッグ、フォルダヘッダと
  「未分類」ヘッダが `DragTarget` で受領。`folderId` を切り替えて `updateEntry` する。Phase 3 で実装済み。
  並べ替え（同一グループ内の順序入れ替え）は未対応で、必要になったら別 ADR で扱う。
- **サイドバーのドラッグ&ドロップ + 右クリック**: Phase 4 で管理画面と同等のドラッグ操作をサイドバーにも
  展開。フォルダタイルへドロップで配下に移動、ランチャーセクション中の「未分類」mini-header へドロップで
  root に戻す。`_LauncherHeader` の右クリックで「新しいフォルダ」「新しいエントリ」のコンテキストメニュー、
  `_LauncherFolderTile` の右クリックで「リネーム / 削除」を表示し、管理画面に行かずにサイドバーから直接
  操作できる。
- **「未分類」というラベル**: 「root」「フォルダなし」「未分類」など候補があったが、ユーザー目線で
  「分類されていない＝未分類」が直感的なのでこれを採用。
