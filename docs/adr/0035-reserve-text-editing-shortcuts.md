# ADR-0035: ⌘C/⌘V/⌘X/⌘A/⌘Z をテキスト編集用に予約し、コマンド割り当て不可とする

- **Status**: Accepted
- **Date**: 2026-05-18

## Context

ADR-0033 でショートカットをカスタマイズ可能にした際、トレードオフとして「ユーザーがあえて ⌘C / ⌘V / ⌘X / ⌘A / ⌘Z を割り当てた場合の OS 標準キーとの競合は検出しない」とした。既定キーはこれら 5 つを避けているが、キーレコーダ上はユーザーが任意のコマンドへ割り当てできる状態だった。

一方で実害が顕在化した: Roola のターミナル（SwiftTerm ネイティブビュー、ADR-0031）には**コピー & ペースト機能が無い**。`TerminalViewDelegate.clipboardCopy` は空実装で、⌘C もどのコマンド・メニュー項目にも結びついていなかった（ADR-0033 の独自 `PlatformMenuBar` が macOS 標準の「編集 > コピー」を置き換えたため）。

ここで「⌘C をコマンドに割り当て可能なまま」にしておくと、コピー & ペーストの居場所を恒久的に確保できない。⌘C/⌘V はテキスト編集の普遍的な操作であり、ユーザー定義コマンドより優先して固定すべきという結論に至った。

## Decision

⌘C / ⌘V / ⌘X / ⌘A / ⌘Z（**`⌘` のみ + 当該キー**の組）を **テキスト編集用の予約コンビ**とし、コマンドへの割り当てを禁止する。あわせてターミナルのコピー & ペーストを実装する。

- **予約コンビの拒否**: `isReservedChord`（`core/keybindings/key_chord_recorder.dart`）を追加し、キーレコーダが予約コンビを検出したら保存をブロックして理由を表示する。⌘⇧C（既定の `copyPath`）や ⌘⌥C（`copyItem`）のように修飾キーが増えたコンビは別物として割り当て可能のまま。
- **コピー & ペーストはコマンドレジストリの外で扱う**:
  - Flutter のテキストフィールドは Flutter 標準のテキスト編集ショートカット（`DefaultTextEditingShortcuts`）がメニューバーに依存せず機能するため、追加実装は不要。
  - ターミナル（SwiftTerm ネイティブビュー）は native の `keyDown` ローカルモニタで素の ⌘C / ⌘V を横取りし、SwiftTerm の `copy:` / `paste:` アクションへ転送する。`clipboardCopy` デリゲートを実装し、選択テキストを `NSPasteboard` へ書き出す。ペーストは SwiftTerm がブラケットペーストモードを含めて処理する。
- 本 ADR は ADR-0033 の「予約しない」というトレードオフ判断のみを更新する（amends）。コマンドレジストリ / ネイティブメニューバーの機構自体は不変のため、ADR-0033 は `Accepted` のまま維持する。

## Why

### なぜコマンドではなく予約にするか

⌘C/⌘V を「コピー / ペーストコマンド」としてコマンドレジストリに載せると、メニューバーの key equivalent はフォーカス非依存で発火するため（ADR-0033）、テキストフィールド編集中の ⌘C がコマンド側に横取りされる。ADR-0033 が ⌘C/⌘V/⌘X/⌘A/⌘Z を既定から外したのと同じ理由で、これらはコマンド機構に載せるべきではない。

代わりに「予約」とし、コマンド機構の**外**で処理する。Flutter テキストフィールドはフレームワーク標準のショートカットでフォーカス時のみ動作し、ターミナルは native モニタでフォーカス時のみ横取りする。どちらもフォーカス依存で、テキスト編集の自然な挙動と一致する。

### なぜ native モニタでターミラルを処理するか

SwiftTerm の `keyDown` はモジュール外から override できず、Roola の `PlatformMenuBar` には標準 `copy:` 項目を載せられない（Flutter の `PlatformProvidedMenuItem` にコピー項目が無い）。既に Shift+Enter（ADR-0032）で使っている `NSEvent` ローカルモニタの仕組みを流用すれば、フォーカス中ターミナルだけを対象に最小実装でコピペを実現できる。

## 代替案

### 代替案 1: ⌘C/⌘V をコマンドレジストリに `copy` / `paste` コマンドとして追加

設定画面で一覧でき統一感はあるが、メニュー key equivalent がフォーカス非依存で発火し、テキストフィールドの編集と競合する。ADR-0033 がこれらを既定から外した判断と矛盾する。**却下。**

### 代替案 2: 予約せず現状維持（ADR-0033 のまま）

⌘C をユーザーがコマンドに割り当てられる状態を許容する。だがそうするとコピー & ペーストの固定的な置き場が無く、ターミナルのコピペを安定して提供できない。ユーザー要望（コピペは C/V に固定）にも反する。**却下。**

### 代替案 3: ネイティブの Edit メニューを Swift 側で注入する

`PlatformMenuBar` が組んだメニューに、Swift 側で標準 Copy/Paste 項目（`copy:` / `paste:`）を足す。OS 標準に最も近いが、`PlatformMenuBar` がメニューを所有・再構築するため注入の維持が難しく、フォーカス非依存発火の問題も残る。**却下。** native モニタ方式を採る。

## Trade-offs

- **⌘X / ⌘A / ⌘Z はターミナルでは未実装** — 予約はするが、ターミナルのネイティブ実装はコピー（⌘C）とペースト（⌘V）に限る。ターミナル出力に対するカット・取り消しは意味が薄く、全選択は将来課題。Flutter テキストフィールドでは標準ショートカットがそのまま機能する。
- **既存の `keybindings.json` に予約コンビが残っていても移行しない** — 本機能は新規で、既定キーに予約コンビは無いため実害はほぼ無い。レコーダが新規割り当てを拒否するのみとする。
- **`charactersIgnoringModifiers` でキー判定する** — ⌘C の判定は keyCode ではなく文字で行う。レイアウト差に強い一方、デッドキー等の特殊配列での挙動は将来検証課題。
