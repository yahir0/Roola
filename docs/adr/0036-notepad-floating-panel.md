# ADR-0036: ノートパッドをワークスペース外のフローティングパネルとして実装する

- **Status**: Accepted
- **Date**: 2026-05-18

## Context

Roola を使う開発者は、作業中にコマンドの控え・一時的なパス・思いついたメモなどを書き留めたい場面がある。現状は外部のメモアプリやエディタへ切り替える必要があり、アプリ内で完結しない。

要件は「ちょっとしたメモ書き」に割り切ったもので、次の範囲とする:

- 本文の入力ができること
- 標準的なテキスト編集ショートカット（コピー / ペースト / カット / 全選択 / 取り消し）が使えること
- 左端に行番号ルーラがあり、行数が分かること
- 配置はワークスペース右下（ターミナルの横）に固定
- ヘッダの設定アイコン横にトグルアイコンを置き、クリックで開閉
- 本文はアプリ再起動後も保持されること

明示的に範囲外とするもの: txt 等へのファイル書き出し、シンタックスハイライト、複数ノート、検索置換、フォント設定。

ワークスペースは 3 ペインタブ式（ADR-0026）で、タブ種別 union（`ExplorerTab` / `TerminalTab` / `GitTab`）はペイン配置・DnD・per-tab 状態・`workspace.json` 永続化と密に結合している。

## Decision

ノートパッドは **ワークスペースのペイン / タブ機構の外側にある独立したフローティングパネル** として実装する。

- **配置**: `WorkspacePage` の `_WorkspaceArea` を `Stack` で包み、`Positioned` でワークスペース領域の右下に重ねる。3 ペインのレイアウト・スプリッタ比率には一切影響しない。固定サイズ（380 × 320）。
- **開閉**: ヘッダ（`MacosWindowAppBar.actions`）の設定アイコン左に `Icons.sticky_note_2_outlined` のトグルボタンを置く。開閉状態はワークスペース内に閉じた一時的な UI 状態のため、Provider ではなく `WorkspacePage`（`HookWidget`）の `useState` で持つ。
- **本文の状態**: `NotepadViewModel`（`Notifier<String>`）が単一の真実。パネルを閉じても破棄されないよう非 autoDispose の `NotifierProvider` とし、再オープン時に最後の本文を復元する。
- **永続化**: `<appSupport>/notepad.json`（`{"content": "<text>"}`）に保存する。差し替え可能性のため `NotepadRepository` interface を置く（他の永続化と同方針）。本文は単一文字列のため DTO ⇄ モデル分離は行わず、リポジトリが文字列と JSON を直接変換する（`locale_settings` の ADR-0034 と同方針）。入力のたびに 300ms の debounce で書き込み、プロバイダ破棄時に未保存分を flush する。
- **起動時読み込み**: 初回オープン時に本文を即表示できるよう、`main()` で同期読み込みし `notepadInitialContentProvider` を `overrideWithValue` で注入する（ワークスペース ADR-0028 / 言語 ADR-0034 と同方式）。
- **行番号ルーラ**: 本文 `TextField` の左に専用の gutter widget を置く。論理行（`\n` 区切り）ごとに 1 番号を表示し、行が折り返しても番号がずれないよう各論理行の高さを `TextPainter` で実測する。`StrutStyle.forceStrutHeight` で 1 表示行の高さを固定し、`RenderEditable` のキャレット余白ぶんを折り返し幅から差し引いて本文と一致させる。
- **ショートカット**: 追加実装はせず、`TextField` 標準のテキスト編集ショートカットに委ねる。⌘C/⌘V/⌘X/⌘A/⌘Z はコマンド割り当て不可キーとして既に予約済み（ADR-0035）なので、アプリのコマンド機構と競合しない。

## Why

### なぜタブ種別ではなくフローティングパネルか

ノートパッドの要件は「右下固定・トグルで開閉」であり、ペイン配置の自由度・DnD・複数枚といったタブ機構の機能を必要としない。`WorkspaceTab` union に `NotepadTab` を加えると、ペイン崩し再フロー・per-tab family・`workspace_layout_dto` のスキーマ migration まで波及し、「割り切ったメモ」という要件に対して構造コストが過大になる。ワークスペース外の独立パネルにすることで、既存のタブ/ペイン/永続化コードを一切変更せずに追加できる。

### なぜ開閉状態を Hook ローカル状態にするか

開閉はそのウィンドウのワークスペース内に閉じた一時的な表示状態で、永続化も他画面との共有も不要。`docs/architecture.md` の「ローカル状態は flutter_hooks」に従い `useState` で持つのが最小かつ適切。一方、本文はパネルの mount/unmount をまたいで保持し永続化もするため Provider に置く。

### なぜ行番号ルーラを自前実装するか

要件はシンタックスハイライト等を含まないため、コードエディタパッケージの導入は過剰（自己完結方針 ADR-0005）。必要なのは「左に行番号」だけで、`TextField` + 実測 gutter で十分に実現できる。

## 代替案

### 代替案 1: `WorkspaceTab.notepad` を追加してタブにする

タブ機構に乗せれば任意ペイン配置・複数ノートが自然に得られるが、`workspace_tab.dart` / `workspace_layout_dto.dart` / `pane_tab_strip.dart` / 崩し再フロー / per-tab family の破棄処理まで変更が波及する。「右下固定・1 枚」の要件に対して不要な機能とコストが大きく、却下。

### 代替案 2: 別ウィンドウ（マルチウィンドウ）で開く

ADR-0012 の別プロセス起動方式に乗せる案。ウィンドウ管理コストが大きく、「ターミナルの横で見ながらメモ」という近接性が失われるため却下。

## Trade-offs

- パネルは固定サイズで、リサイズ・移動・ドッキング位置変更はできない。要件が「割り切ったメモ」のため許容する。将来必要になれば本 ADR を継ぐ形で拡張する。
- 行番号ルーラは `RenderEditable` のキャレット余白（`_kCaretGap` + `cursorWidth`）に依存して折り返し幅を合わせる。Flutter SDK 側がこの内部定数を変えた場合、折り返し位置がわずかにずれ得る（FVM でバージョン固定のため実害は小さい）。
- 本文は 300ms debounce で保存するため、最後の入力から 300ms 以内にプロセスが強制終了されると末尾の編集が失われ得る。通常終了経路ではプロバイダ破棄時の flush で取りこぼさない。
