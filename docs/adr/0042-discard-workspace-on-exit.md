# ADR-0042: アプリ終了時にワークスペースを破棄し、起動は既定 seed で始める

- **Status**: Accepted
- **Date**: 2026-05-21
- **Supersedes**: ADR-0028（ワークスペースレイアウトの永続化とターミナル再 spawn）

## Context

ADR-0028 では `workspace.json` にレイアウトを保存し、次回起動時に復元する仕様を
採用していた。エクスプローラタブはカレントパスを、ターミナル / Skill タブは作業
ディレクトリと `LauncherAction` を保存して PTY を再 spawn する。

運用してみて、次の挙動がはっきり不快であるとわかった。

- 終了時には「セッションが残っているけど終了しますか？」と尋ねるのに、次回起動
  すると同じタブ群が復活する。終了確認の意味合いと矛盾する
- 復元時にターミナル / Skill タブが自動で PTY を spawn するため、`RunCommandAction`
  系の Skill は **再起動するだけで意図せずコマンドが再実行される**。Skill の動作
  によっては副作用（ファイル書換え・外部 API 呼び出し）が走る
- ADR-0028 自体が「ターミナルの履歴は失われる」「再 spawn になる」と Trade-off
  として明示しているが、その許容範囲をユーザー UX が超えている

「閉じたら閉じる、開き直したらまっさら」のほうが、本アプリの利用形態（ワン
ショットで起動して Explorer / ad-hoc ターミナルを使う）と一致する。

## Decision

ワークスペースレイアウトを永続化せず、**毎回既定 seed で起動する**方針に転換する。

### 起動

- `main.dart` から `workspace.json` のロードを削除する
- `workspaceInitialLayoutProvider` の override も削除し、`seedDefaultWorkspace()`
  をそのまま初期値として使う
- 旧バージョンで残った `workspace.json` が存在する場合は、起動時にベスト
  エフォートで削除する（マイグレーション）

### 実行中

- `workspaceProvider._apply` から永続化呼び出し（`_persist`）を削除する
- `WorkspaceRepository` / `WorkspaceRepositoryImpl` 自体は将来の再導入に備えて
  コードベースに残すが、参照は無くなる
- `AppPaths.workspaceFile` も保持する（マイグレーション削除で参照するため）

### 終了

- 既存の `WindowCloseGuard` 経由のセッション cancelAll → `windowManager.destroy()`
  の流れはそのまま維持する。レイアウトを書き戻さなくなるため、終了は単に
  「全部閉じる」だけになる

## Why

- **再 spawn の副作用が UX を壊している**: ADR-0028 の段階では PTY 再 spawn は
  「素のシェルなら実用上同じ」と評価していたが、Skill / `RunCommandAction` の
  再実行が実害になっている。Trade-off が想定より重い
- **終了確認との一貫性**: 「終わるか？」と問うた以上、次回起動は素のスタートに
  戻すのが期待挙動。永続化してしまうと終了確認が形だけのものになる
- **失う価値が小さい**: 永続化される情報はカレントパスとペイン比率程度。
  毎回手動で開き直すコストは小さい。むしろ「前回の状態に縛られない」自由度の
  ほうが、汎用ランチャーという位置付け（ADR-0014 / ADR-0016）に合う

## 代替案

### 代替案 1: ターミナル / Skill タブだけ復元しない

エクスプローラタブ・ペイン比率のみ復元し、ターミナル / Skill タブは捨てる。

- 「どこまで復元するか」の境界線がユーザーから見て分かりづらい
- 「Skill だけ走らないが、ターミナルは spawn する」など細分化を始めると ADR が
  肥大する
- 後で「明示的な復元」が欲しくなれば、設定で off → on 切り替えのほうが素直
- 却下。

### 代替案 2: 復元のオン / オフを設定で切り替え

ユーザー設定で「前回のワークスペースを復元」のトグルを設ける。

- 設定 UI と永続化スキーマが増える。当面は要らない
- 復元したい要望が積み上がってから追加すれば良い
- 却下（後追いで再導入する余地は残す）。

### 代替案 3: 復元するが、ターミナル / Skill タブは「停止状態」で復活させる

タブは復活させるが PTY は spawn せず、ユーザーが明示的に「起動」ボタンを押す
までは動かさない。

- セッション復元の挙動として悪くないが、停止タブを表す UI と起動操作の追加
  実装が必要で、永続化を残し続けることになる
- 「閉じたら閉じる」というメンタルモデルからは遠ざかる
- 却下。

## Trade-offs

- **ペイン比率・ペインのタブ並びが毎回リセットされる**: 既定 3 ペイン（$HOME ×2 +
  ターミナル）に戻る。ユーザーが好む配置にしていた場合、毎回作り直す必要がある
- **エクスプローラのカレントパスも戻る**: 直前にいた深い階層は失われる。お気に
  入り（既存機能 / ADR-0029）でカバーする想定
- **将来「復元したい」要望が出たら再導入が必要**: 代替案 2 の方向で別 ADR を
  立てる
