# ADR-0026: 3 画面タブ式ワークスペースの採用

- **Status**: Accepted
- **Date**: 2026-05-15

## Context

`/explorer`（`ExplorerPage`）は ADR-0014 で単一ペインのメイン UI になって以来、
`explorerSelectionProvider`（`directory` / `entrySession` / `adhocSession` の sealed
union）でディレクトリ一覧かセッション 1 件のどちらかを排他表示するだけだった。

このモデルでは、

- 2 つのディレクトリを並べて参照できない（コピー元 / コピー先を行き来する作業が苦しい）
- ファイル操作をしながらターミナルを併用できない（selection が片方を潰す）

ターミナルランチャーアプリとして、複数エクスプローラ + ターミナルの同時利用は
中核的なユースケースであり、単一ペイン前提のままでは頭打ちになっていた。

## Decision

`/explorer` を **3 ペインスロット（`topLeft` / `topRight` / `bottom`）× タブ群** の
ワークスペースに刷新する。

- 各ペインスロットはタブ群 + アクティブタブを持つ。タブは種別固定（エクスプローラ /
  ターミナル）
- 初回起動 / 永続データ無し時は既定 3 ペインを seed: 上 2 つがエクスプローラ($HOME)、
  下がターミナル(素のシェル @ $HOME)
- スロットのタブが 0 になると崩し再フローし、コンテンツが 1 スロットだけになれば
  従来どおり単一ペイン全画面で描画する
- 上下・上段左右の 2 本のスプリッタでリサイズできる
- 表示単位を「タブ」にするため `explorerSelectionProvider` は廃止する

per-tab 状態の実現方法は ADR-0027、永続化は ADR-0028 に分ける。

## Why

- **複数並行参照が本質的要求**: エクスプローラ 2 面 + ターミナルの構成はファイル
  操作の主要動線。selection 1 個で潰し合うモデルの延命では解決しない
- **崩し再フローで「単一ペインに戻れる」**: 3 ペインを強制せず、タブを閉じれば
  従来の単一ペイン体験に自然に縮退する。新レイアウトが負担になる場面で逃げ場がある
- **タブ種別を固定にした**: 旧 `explorerSelectionProvider` は 1 ペイン内で
  「ディレクトリ ⇄ セッション」を動的に切替えていたが、これはタブ × ペインの
  二次元モデルと噛み合わない。タブの種別を固定し、切替えではなくタブ追加で表現する

### 代替案 1: `explorerSelectionProvider` を family 化して残す

ペイン ID をキーに selection を family 化し、各ペインが従来どおりディレクトリ /
セッションを動的切替する案。

- 「戻る矢印で session→directory に戻す」等の動的切替ロジックがタブモデルと二重に
  なり、かえって複雑化する
- タブの並べ替え・ペイン間移動を表現する単位が無い
- 却下。

### 代替案 2: 既存の単一ペインを残し、別ルートで分割ビューを追加する

`/explorer` はそのまま、`/workspace` を新設して併存させる案。

- 同じエクスプローラ機能が 2 系統に分かれ、保守が二重化する
- メイン UI は 1 つであるべき（ADR-0014 の方針）
- 却下。

### 代替案 3: ペイン数・配置を任意グリッドで可変にする

IDE のような自由分割。

- スプリッタ・ドロップ判定・永続化スキーマが大幅に複雑化する
- 「上 2 + 下 1」が要求の実体であり、固定 3 スロットで十分
- 却下（将来必要なら別 ADR）。

## Trade-offs

- **`explorerSelectionProvider` 廃止の波及が広い**: `explorer_page` / `sidebar` /
  `node_tile` / `path_bar` / `launcher_actions` / `session_view` / run・adhoc VM の
  `onClosed` 経路に及ぶ。状態基盤フェーズで一括移行する
- **3 分割時の各ペインが狭い**: ターミナルや一覧の表示領域が小さくなる。スプリッタ
  と崩し再フローで緩和する
- **レイアウトモデルの保守コスト**: スロット × タブの二次元状態を扱う
  `workspaceProvider` が増える。崩し再フローは純粋関数に切り出してテスト可能にする

## Update (2026-05-18): セッションヘッダの停止ボタンを廃止

`session_view.dart` のセッションヘッダに置いていた停止ボタン（`Icons.stop` →
`runner.cancel()` で PTY を SIGTERM）を削除した。

- 停止ボタンは「実行中コマンドの中断」ではなく **PTY セッション丸ごとの終了** で、
  素のシェル（`OpenHereAction`）では常時表示されていた。ADR-0016 でランチャーが
  汎用ターミナル化し「run」が常駐シェルになった結果、「run をキャンセルする」と
  いう操作モデル自体が実体と噛み合わなくなっていた
- コマンドの中断はターミナル内で Ctrl+C を打てば PTY に伝わる。セッションの終了は
  タブストリップの × で行える（× → `Workspace.closeTab` → provider invalidate →
  `runner.dispose()` で PTY kill）。停止ボタンは後者と重複していた
- よって「ターミナルの終了はタブの × に一本化する」という本 ADR のタブモデルに
  沿って停止ボタンを撤去した。`SkillRunCancelled` 状態・`restart`（再実行）ボタン・
  runner の `cancel()` メソッドは残す（`cancel()` はウィンドウクローズ時の
  `ActiveSessions.cancelAll()` が一括 kill で使用するため）
- l10n キー `sessionCancelTooltip`（ja / en）も未使用となり削除した
