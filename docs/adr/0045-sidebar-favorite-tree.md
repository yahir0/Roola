# ADR-0045: サイドバーのお気に入りを Win2000 風ツリーで展開する

- **Status**: Accepted
- **Date**: 2026-05-21

## Context

サイドバーのお気に入り（[ADR-0029] でフォルダ単位のグループ化済み）は、
クリックすると当該パスへ遷移する 1 行だけのタイル。配下のディレクトリ構造
は本体エクスプローラに切り替えてから初めて見える。

頻繁に潜るプロジェクト（`src/` 配下や `docs/` 配下など）では、サイドバーから
直接「2 階層下のフォルダ」へ飛べると navigation が高速化する。Win2000 時代の
エクスプローラ左ペインのようなツリー展開がそのモデルに近い。

[ADR-0029]: 0029-favorite-folder-grouping.md

## Decision

各お気に入りに **chevron** を追加し、クリックで配下の **サブディレクトリ**
を再帰的に展開できるようにする。展開状態はサイドバーが Hook（`useState<Set<String>>`）
で保持し、**永続化しない**（セッション内のみ・ランチャー / お気に入りフォルダの
展開状態と同じ扱い）。

### お気に入りフォルダ（ユーザー命名グループ）との UI 差別化

両者とも「chevron で展開できる行」になるが、機能が違うので一目で区別できる
よう **chevron の太さ** と **leading icon の系統** で差別化する：

| | お気に入りフォルダ（[ADR-0029]） | お気に入り + ツリー（本 ADR） |
| --- | --- | --- |
| chevron | `arrow_drop_down / arrow_right`（太い三角・standard 18px） | `expand_more / chevron_right`（細いシェブロン・small 16px） |
| leading icon | `Icons.folder_outlined`（Material outlined） | `PolarisTypeIcon`（Polaris 幾何モノライン / D10） |
| 意味 | ユーザー命名のラベル / グループ | 実 filesystem のパス |
| 子要素 | favorites（人が登録したもの） | サブディレクトリ（filesystem から自動取得） |
| インデント | 配下 favorite は 1 段 | 階層ごとに 12px ずつ深く |

太さ違いの chevron と異なる leading icon 系統で、「これはユーザーが命名した
箱」「これは filesystem」が読み取れる。

### スコープ

- **ディレクトリのみ表示**: ファイルは出さない（サイドバーは navigation 補助）
- **隠しディレクトリ（`.git` 等）も出す**: 本体エクスプローラと整合
- **永続化なし**: セッション内のみ。アプリ再起動で初期状態に戻る
- **filesystem 監視なし**: 展開時に都度 listing を読み直す（`Provider.autoDispose`
  で畳むと破棄）。ターミナル等でファイルが追加されたらユーザーが畳んで開き直す

### クリック挙動

| クリック位置 | 動作 |
| --- | --- |
| chevron | 展開トグル |
| アイコン / ラベル | フォーカス中エクスプローラを当該パスへ遷移（既存挙動） |

chevron に独立した `GestureDetector` を被せて hit target を分離する。
外側の `InkWell`（navigate）とはネストするが、より specific な内側が
gesture arena で勝つ。

## Why

- **navigation の高速化**: 「お気に入り→ Explorer に切替→深く潜る」を、
  サイドバーから 1 操作で飛べるようにする。Win2000 はこの操作モデルで広く
  普及した
- **既存 chevron 流用で十分**: 新たな disclosure pattern を導入する必要は
  なく、お気に入りフォルダで使っている chevron の **太さと icon 系統** を
  振り分けるだけで読み取れる
- **永続化しないのは妥当**: 展開状態を全パス分永続化すると、ストレージが
  パスゴミで膨れる。畳むのはコスト 0、開くのは listing 1 回なので、セッ
  ション内記憶だけで実用十分
- **FSEvents 監視を入れない**: 監視は本体エクスプローラ向け（[ADR-0041]）の
  ローカルなもの。サイドバーのツリー全パスを監視するのは複雑度に見合わない。
  navigation 補助なので多少 stale でも問題ない

[ADR-0041]: 0041-realtime-fs-watch.md

## 代替案

### 代替案 1: 展開状態を `ExplorerSettings` に永続化する

- アプリ再起動後も展開状態が残る
- 「ユーザーがどこを開いたか」がストレージに蓄積される → プライバシ的にも
  軽くノイズ
- 「保存・読み込み」の同期問題（パス削除時の cleanup 等）が追加で増える
- セッション内のみで十分との判断で却下。要望が出れば再検討

### 代替案 2: 別画面（フォルダピッカーダイアログ）でツリーを出す

- サイドバーが膨らまないが、ワンクリックで深部に飛ぶ体験は失われる
- 却下。Win2000 風＝**サイドバー内**で完結することが要件

### 代替案 3: 展開時に FSEvents 監視を張る

- 「他プロセスでフォルダが作られたら自動反映」できる
- 展開中のパス N 個ぶん監視を張る複雑度。サイドバー目的でそこまで必要ない
- 却下。畳んで開き直せば再読込される

### 代替案 4: ツリーノードを Win2000 風の点線で繋ぐ

- 親子関係が視覚的に分かりやすい
- Polaris D3（装飾線禁止・トーン差と 1px ラインのみ）と相反
- インデントだけで十分。却下

## Trade-offs

- **隠しディレクトリも出す → ノイズ**: `.git` `.vscode` `.DS_Store` 等で
  ツリーが膨らむ場合がある。実害が顕在化したら filter を後付けする
- **listing は同期 IO**: 展開した瞬間に `Directory.listSync()` が走る。
  大量のサブディレクトリを持つパスではフレーム落ちの可能性。実害が出れば
  async + AsyncValue 化を検討
- **指定パスが消えた場合**: ローダーが空リストを返すので、展開しても何も
  出ない。chevron は出たままになる（誤クリック時の挙動は無害）
