# ADR-0021: エクスプローラの操作モデルをダブルクリック化 + CC でパスコピー

- **Status**: Accepted（`C` `C` 連打部分は ADR-0033 により Superseded。ダブルクリック式の操作モデルは有効）
- **Date**: 2026-05-14

## Context

これまでのエクスプローラはシングルクリックでディレクトリ遷移 / ファイル open する設計だった
（ブラウザ風）。しかしユーザーから「ファイルマネージャに近い操作感が欲しい」「クリックで選択して
何かをしたい場面が多い」というフィードバックがあった。具体的には:

- フォルダパスをコピーしたいことが多いが、選択状態を経ずすぐ遷移してしまう
- ファイルを開かずに「これだ」と確認したいことがある
- Finder / Explorer / ファイル系ツールに慣れた手だと「クリック = 選択」「ダブルクリック = 実行」が
  デフォルトの期待値

合わせて、選択中アイテムのパスを素早くクリップボードへ取り出す手段が欲しい。

## Decision

操作モデルを次のように変更する:

| 操作 | 旧 | 新 |
|---|---|---|
| シングルクリック | 遷移 / open | **選択（ハイライト）** |
| ダブルクリック | （なし） | **遷移 / open** |
| `C` × 2 (500ms 以内) | （なし） | **選択中アイテムのパスをクリップボードへコピー** |
| 右クリック | コンテキストメニュー | （変更なし） |
| マウス進む/戻る | 履歴ナビゲーション | （変更なし） |

### 実装方針

- 選択状態は `ExplorerItemSelectionNotifier`（`NotifierProvider<String?>`）で 1 アイテム単位で保持。
  ディレクトリ遷移時は自動でクリア（`_DirectoryListing` 内の `ref.listen`）
- `_DirectoryTile` / `_FileTile` の `InkWell.onTap` を「選択」、`onDoubleTap` を「遷移 / open」に
  振り替え。選択中アイテムは `primary` 色 16% alpha の背景でハイライト
- 「上の階層へ」タイル（`ExplorerParentDropTile`）はナビゲーション専用 widget なのでシングル
  クリックのまま維持（戻るボタンと同等の扱い）
- CC 連打は `_DirectoryListing` の `Focus(autofocus: true, onKeyEvent: ...)` で検出。`useRef`
  で前回 C 押下時刻を保持し、500ms 以内の 2 回目で `Clipboard.setData` + `SnackBar` 通知
- 設定画面に「キーボードショートカット」セクションを追加し、操作モデルを明示

### 500ms / Vim 風 `yy` 風の理由

Vim の `yy`（行コピー）と同じく、同一キーの連続でアクションを起動する慣習はエディタ系ユーザーに
馴染みがある。間隔は Flutter のデフォルト double-tap タイムアウト（~300ms）よりやや長めの 500ms
にして、急がず確実に 2 連打できる感覚にする。C 以外のキーが間に挟まれたらシーケンスはリセット。

## Why

### なぜダブルクリック式か

- ファイルマネージャ系ツール（Finder / Explorer / VS Code エクスプローラ）の標準的な期待値
- シングルクリック = 選択 が可能になることで、パスコピー / プロパティ確認 / 削除 / リネームなどの
  「選択 → 操作」フローが自然に成立する
- 「すぐ遷移して欲しい」場面はダブルクリックで失われた効率を取り戻し、選択フローでの活用シーンが
  増える方が総合で得

### なぜ CC（C 連打）か

- パスコピーは頻度が高いので片手で完結するショートカットが欲しい
- Cmd+C は意味的に「選択中テキストをコピー」と被るので避けたい
  （将来テキスト選択も入れたとき衝突する）
- Vim 系の `yy` / `dd` / `cc` パターンが「同じキー連打 = 同名対象に作用」の慣習として根付いている
- 1 キーだと意図しない連打や typing 中の誤爆が出やすい。2 連打にすると暴発しにくい

### なぜ「上の階層へ」だけはシングルクリックのまま

ナビゲーション専用 UI で、選択する対象として扱う意味が無い（フォルダノードのコレクションではなく
「機能ボタン」）。AppBar の ← → と同じく「クリックしたら動く」が自然。

## 代替案

### 代替案 1: シングルクリック維持 + Cmd+C / 別ショートカットでコピー

- 操作の癖を変えずに済むが、「クリック → 即遷移」のため選択フローが存在せず、コピー対象を選ぶには
  わざわざマウスをホバーしておくなど不自然
- 却下: 選択 = ハイライトの状態がないと、コピー対象の事前確認ができない

### 代替案 2: ホバー中アイテムを暗黙の選択にする

- マウスホバーだけで「これがコピー対象」と扱う案
- 触らずにホバーしただけで状態が変わるのは事故の原因になりやすい
- キーボードでの focus がホバーと食い違うと混乱
- 却下

### 代替案 3: Cmd+C で選択 / 単押で発火

- 既存のテキスト UI と挙動が混在する。Cmd 系は「選択中テキストの clipboard」と Roola 独自の
  「選択中ファイルパスの clipboard」が両立しなくなる
- 却下: 衝突回避のため自前キー (CC) を採用

## Trade-offs

- **学習コスト**: 既存ユーザーはシングルクリック遷移に慣れている。設定画面に解説を載せて、
  「設定 → キーボードショートカット」で初見でも見つけられるようにする
- **誤入力時の挙動**: 選択中にうっかり C を 2 回押すと意図せずパスがクリップボードに乗る。
  上書きの副作用は無視できるが、SnackBar で「コピーしました」を明示するので、ユーザーが気付く
- **テキスト入力欄での挙動**: パスバー等で C を打っているときは Focus が子に移っているので、本
  ハンドラは発火しない（`Focus` 仕様に基づく自然な挙動）
- **キーボード focus の保持**: タイルクリックで子 InkWell が一時的に focus を取るが、`Focus`
  ツリーをたどって本ハンドラまで来るので動作する（Shortcuts と同じ「Focus traversal で
  bubble-up」原理）
