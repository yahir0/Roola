# ADR-0038: Polaris デザインシステムを採用する

- **Status**: Accepted
- **Date**: 2026-05-19

## Context

これまでの Roola は Flutter Material 3 のデフォルトに、ADR-0020 で導入した
「Win10/11 風フラット実用テーマ」（`lib/app/theme.dart` の `AppTheme` /
`LogoTheme`）を乗せた構成だった。角丸 2px・elevation 0・`VisualDensity.compact`・
ライト/ダーク 2 テーマ。

ユーザーから「Flutter（Material Design）のモバイル感・Android 感を完全に排除
したい」「ADR-0020 の Win11 フラットでも Flutter らしすぎる」というフィード
バックが入った。検討用プレビュー（`lib/dev/cockpit_preview.dart`、独立した
`main()` を持ち既存アプリに影響しない）を `flutter run -t` で起動し、実物を見
ながら 17 回反復した。言葉のニュアンス（「コックピット」「アメリカ感」など）は
誤解を生みやすく、実物を見て反応する方が速く正確だったため。第 17 稿が最終形。

この反復で「コックピット」という当初の表現は実態と合わなくなり（装飾的な
モチーフはすべて却下された）、最終的に体系は **機能主義（functionalism）** へ
着地した。ディーター・ラムス → ジョナサン・アイブの系譜（「より少なく、しかし
より良く」）。

## Decision

**独自デザインシステム「Polaris」を採用し、ADR-0020 のテーマを全面置換する。**

Polaris の全デザイン値は `PolarisTokens`（`ThemeExtension`）に集約し、UI
コンポーネントはそこを経由して参照する。色・角丸・余白のハードコードを禁止する。

主要な決定（検討プレビュー反復で固めたもの）:

### D1. 思想を「機能主義」に置き、体系を Polaris と名付ける

デザインシステムは特定のビジュアルでなく「機能主義」という思想を基盤とする。
思想は不変、ビジュアル（色・寸法等）はそこからの導出物で、機能的に優れる選択が
あればビジュアルは差し替えてよい。体系名は Polaris（アプリ名 Roola＝移動呪文
ルーラに対する「移動のための導きの星」。不動の原理から可変のデザインが導かれる
構造を一語で表す）。却下案: 質感由来の名（Billet/Instrument/Gauge）、数学由来の
名（Axiom/Invariant/Basis）。

### D2. ダーク専用にする

ライト/ダークの 2 テーマをやめ、ダーク専用にする。「沈んだ計器ディスプレイ」
「暗い地のかっこよさ」はライトでは再現できず別物になるため。トークン構成も
ダーク前提で単純化できる。却下案: ライト併存（世界観が割れる・検証コスト倍）。

### D3. 地はグラファイト。well/bg の 2 トーン階層

地は純黒でなく僅かにリフトしたグラファイト。`well`（最も暗い＝沈んだ計器
ディスプレイ＝ファイル一覧）と `bg`（一段明るい＝筐体の枠＝トップバー/サイド
バー/ステータスバー）の 2 トーンで「ベゼルに嵌った画面」というマテリアリティを
出す。影は使わず、トーン差と 1px ラインのみで面を分ける。純黒一枚は「素材の
ない空洞」で初見ユーザーに怪しさを与え、リフトしすぎるとハッカー的な暗い
かっこよさが死ぬ。「物体感」は明るさでなく色温度・筐体上端の 1px ハイライト
（`topEdge`）・可読なラベルで出す。却下案: 純黒一枚、グラファイトまで明るく
リフト、Nord/Gruvbox/Dracula のシンタックスパレット（「ソフトのテーマ」に
見えて機材感が出ない）。

**改訂（ターミナル向けに 3 トーン階層へ拡張）**: 当初の 2 トーン構成では
ターミナルの居場所が無かった。Explorer / Git タブのような「計器を読む」面と、
ターミナルの「機械を叩く」面は機能が異なる（前者は受動的なサーベイ、後者は
能動的なタイピング）。前者を well にする一方、後者を bg にすると「サイドバー
と同色＝ナビと同列」と読めてしまい機能的に嘘になる。逆に後者を well にすると
計器の語彙を機械の操作面へ流用することになり、メタファがブレる。

そこで 3 トーン階層に拡張する: `machine`（機材本体＝ターミナル・ウォーム
グラファイト）・`well`（計器ディスプレイ＝ファイル一覧・クールグラファイト）・
`bg`（筐体の枠）。カーコックピットの「機械本体／メーター類／ダッシュボード枠」
に対応する。

`machine` と `well` の差別化は **深さでなくリフト幅** で行う。`machine` は
`bg`（`#121317`）直下まで持ち上げたクールグラファイト `#101115` とする。これは
D3 の純黒回避原則「純黒一枚は素材のない空洞で高級感が抜ける」と整合する:
`machine` を `well` より純黒寄りに深掘りすると終端が空洞化して「機材の手応え」
が逃げる。かつ広い暗面（ターミナルは内容が疎で大きい）が「沈黙の void」として
圧をかける。実機材の操作面（コンソール / ステアリングコラム）もシャシ表面と
同じ高さで、計器類だけが沈むのが自然。

色温度シフト（ウォーム寄り `#0E0D0B`）も検討したが、Roola 全体が cool で統一
されている中で 1 か所だけ warm にすると「素材が違う面」として孤立して見える
ことが実機検証で判明したため撤回。クール統一を維持してリフト幅だけで差別化
する：

- `well` ＝精密計器（読むもの・沈み込み）
- `machine` ＝操作面（叩くもの・シャシ表面と同じ高さ）

機能差は「沈み込み有無」で表現する。

ターミナルは計器ディスプレイパネル（well + 角丸 + インセット）ではなく、
ヘッダ（bg）と機材面（machine）を 1px 継ぎ目で分けるだけのフラットな構成で
描く。業界通例（iTerm2 / Warp / VS Code Terminal 等のフルブリード）と整合
しつつ、Polaris のトーン階層の中に位置づけられる。

### D4. アクセントは暖色ゴールド 1 色

アクセントは `#D0A341`（削り出しの真鍮・ゴールド）1 色のみ。選択・現在地・
フォルダ等、機能的に意味のある箇所に限定使用する。地（グラファイト）が寒色
なので、アクセントも寒色だと全体が無機質・冷たくなる。冷たい筐体に暖色
アクセントで「生きた高級感・温度差」が出る。一度アイスシアン（`#48C9DE`）に
決めかけたが、全体が寒色で無機質になる点が「しっくりこない」とされ、
ICE/AZURE/GOLD/VIOLET を並べて比較し GOLD を採用。却下案: シアン（寒色一辺倒）。

**改訂（アクセントを選択可能にする）**: 「同時に複数のアクセントを使わない
（＝1 色だから効く）」原則は維持したまま、その 1 色をユーザーが切り替えられる
ようにした。選択肢は **ゴールド `#D0A341`（既定）** と **アイスシアン
`#48C9DE`**（着想段階で検討した寒色アクセント）。設定は `AppearanceSettings`
に `PolarisAccent` として永続化し、`AppTheme.polaris(accent:)` がトークンの
`accent` を差し替える。「画面内に複数アクセントを併用しない」制約は不変で、
切り替えても常に 1 色。

### D5. 信号色 green/amber/red を Git 状態で使う

green=新規 / amber=変更 / red=コンフリクト を、ファイル行左の状態マーカーと
ステータスバーの Git サマリで使う。意味専用。「増やしていい色は意味を運ぶ色
だけ」。アクセントがゴールド（暖色）になったため git「変更」の amber と色相が
近い。git「変更」の信号色は、ゴールドと十分に区別できるトーンへ振り直す。

### D6. 角は機械加工 R=4px、全寸法は 4px グリッド

角丸は 0px でなく 4px（`PolarisTokens.radius` の 1 トークンで全体を統制）。
全寸法（行高・バー高・余白）を 4px の倍数に乗せる。ファイル一覧は 8px の
インセットを取り、R=4px・1px ボーダーの「計器ディスプレイパネル」として
ベゼルに嵌め込む。完全な 0px のナイフエッジは安っぽく不安定。4px グリッドの
厳密さ＝精密さそのものが機械の質感になる。

当初は R=2px としたが、設定画面など大きめのパネルでは 2px が実質直角に
読めて硬すぎたため、4px へ引き上げた。半径は `PolarisTokens.radius` の
単一トークンに集約し、ハードコードした `BorderRadius.circular(2)` は全廃した。

### D7. アニメーション 0ms（即時フィードバック）

ink splash・ホバーのフェード・ページ遷移アニメ・スクロールのバウンス/グローを
すべて排除。状態変化は次フレームで即座に。フェードやバウンスは「Flutter らしい
柔らかさ＝モバイル感」の発生源。`splashFactory: NoSplash`、ボタンの
`animationDuration: Duration.zero`、`PageTransitionsBuilder` で遷移を即時化、
`ScrollBehavior` でオーバースクロール表現と慣性を排除（`ClampingScrollPhysics`）。

### D8. macOS タイトルバーを隠し、トップバーに統合

`window_manager` の `TitleBarStyle.hidden` でタイトルバーを隠し、自前のトップ
バーをタイトルバー兼ドラッグ領域（`DragToMoveArea`）にする。左端は信号機ボタン
の領域（約 78px）を確保。アプリ全体を「1 個の機械」に見せる。

### D9. タイポグラフィ: SF Pro（UI）+ SF Mono（データ列）

クローム（ラベル・ファイル名・本文）は OS 標準のプロポーショナル（SF Pro）。
データ列（perm / size / 日時 / パス）は等幅（SF Mono、フォールバック Menlo）。
日本語は Hiragino Sans へフォールバック。本文・ラベルのウェイトは **w500 基準**、
ロゴワードマークのみ w600。全面等幅は「ターミナル/コードエディタの安っぽい
字面」になり却下。

**ウェイトの改訂**: 当初は「Flutter macOS は native のような stem darkening を
せず暗い地で文字が痩せて見える」ことを理由に w600 を基準にしたが、実機で
小さい字 × w600 がずんぐり・丸文字に見え、特に日本語（Hiragino Sans の太め
ウェイト）でスタイリッシュさを欠いた。精密さはウェイトでなくサイズ・トラッ
キング・トーン階層で出す方針に改め、基準を w500 に引き下げた。

### D10. アイコンは幾何学モノラインの自作描画

フォルダ／ファイル／山括弧などのアイコンは `CustomPaint` で幾何学モノライン
（ヘアラインのストローク）を自作描画する。既製アイコン（Material/Cupertino）は
consumer 向けで丸み・親しみが強く、精密な計器 UI の中で温度が合わない。

### D11. 状態インジケーターは「表示灯」型

ステータス点（Git 状態・電源灯・Ready 表示等）は単色の塗り点でなく、同色の
ヘアラインのリング（淡）＋塗りのコア（濃）の同心円＝「ベゼルに嵌った表示灯」
として `CustomPaint` で描く。単色のベタ点は安っぽい。

### D12. 選択モデル: 主選択のみフルアクセント点灯

ファイル一覧の選択行は控えめな塗り（`surfaceHi`）。複数選択時、主選択（アンカー
＝最後に選んだ 1 つ）だけがフルアクセント色で点灯（行名・アイコン・左 2px バー）。
キーボードフォーカスは選択と一体化し、別表示しない。全選択行をフルアクセントに
すると画面がアクセントだらけになり信号性が薄れる。

### D13. 装飾モチーフは一切足さない

「機械っぽさ」を出すための applied な装飾（四隅のレジストレーションマーク、選択の
照準ブラケット、特徴的書体の引用等）は使わない。機械感は構造・精密さ・抑制
（トーン階層・グリッド・1 アクセント）からのみ滲ませる。この原則自体が Polaris の核。

### D14. 背景の外観機能を Polaris と整合させ、透過 / 不透明の 2 択に絞る

Roola には Polaris 以前から「外観」機能（アプリ背景を 透過 / 単色 / 画像 /
グラデーションに切り替える）があった。Polaris はダーク専用（D2）・グラファイト
筐体（D3）を視覚的アイデンティティとするため、背景を任意の単色・画像へ差し替える
モードはこの思想と相反する。外観モードを `opaque`（不透明なグラファイト筐体）と
`transparent`（筐体ごと半透明にし背後のデスクトップを透かす）の 2 択に絞り、
`solid` / `image` / `gradient` を廃止する。旧設定値は `opaque` へフォールバック
移行する。単色・画像はグラファイト筐体の一貫した視覚言語を壊すため廃止し、透過は
「グラファイトが薄くなって背後が透ける」だけで世界観を壊さず、背後のデスクトップを
見るという機能的価値もあるため残す。却下案: 全モード維持（グラファイト筐体と相反）／
外観機能の全廃（透過は機能的価値があり惜しい）。

透過は単一の `Opacity` で UI ツリー全体を一括合成して実現する。Polaris の各
サーフェスは不透明グラファイトを全面に塗るため、面ごとに半透明化するとトーン階層
（`well` / `bg`）が重なる箇所でアルファが二重掛けされ濁る。`Opacity` は子を一旦
合成してから 1 度だけアルファを掛けるため、トーン階層を保ったまま均一に透ける。
`_AppearanceLayer` は両モードとも不透明グラファイトの基底層を必ず敷く（自前で
背景を塗らない画面 — ターミナルのネイティブビュー周り・設定画面 — が透過
ウィンドウ素通しの透明な穴になるのを防ぐ）。この決定は、Polaris 移行で全
サーフェスが不透明グラファイトを塗るようになり背面の外観レイヤーが隠れて
透過・単色が機能停止していた不具合の修正にあたり、外観機能と Polaris の両立を
見直した結果である。

## Why

「Flutter らしさ／Material のモバイル感」の正体は、強めの角丸・surfaceTint と
elevation の浮き・ink ripple の弾力・柔らかいページ遷移である。ADR-0020 は角丸と
ripple を削ったが、ライト/ダーク併存・Material の ColorScheme をそのまま使う構成・
標準アイコン・標準インジケーターは残しており、「Flutter らしさ」が抜けきらなかった。

Polaris はこれを思想から作り直す。デザインシステムを特定のビジュアルでなく
「機能主義」という不変の思想に置き、色・寸法・素材表現はそこからの導出物とする。
体系名 Polaris（不動の基準星）が、この「不変の原理＋可変のデザイン」という構造
そのものを表す。

### 設計プロセスを記録に残す理由

仕様を机上で固めず、独立した検討用プレビューを実機で 17 回反復した。この手法
自体が有効だった（言葉のニュアンスより実物が速く正確）ため、後続の大きな視覚
変更でも同じ手法を採ることを推奨する。

## Trade-offs

- **全コンポーネント横断の大規模変更**: `PolarisTokens` を先に確立し、コンポー
  ネントは領域ごとに段階移行する。各段階で `flutter analyze` を通す。
- **ダーク専用にしたことで明所での視認性・好みに対応できない**: 機能主義の純度を
  優先する明示的な決定（D2）。将来要望が出れば別 change で扱う。
- **ゴールドアクセントと git「変更」amber の色相衝突**: git「変更」信号色を
  ゴールドと区別できるトーンへ振り直す（D5）。
- **Flutter macOS のテキスト描画は native と差があり痩せ/滲んで見えうる**:
  ウェイト w500 基準で字面を確保する。w600 まで上げると小さい字でずんぐり・
  丸文字に見えるため避ける（D9）。
- **透過モードの `Opacity` がプラットフォームビューに効かない可能性**:
  Flutter の `Opacity` はネイティブビューに追従しないことがあり、透過モードで
  SwiftTerm のターミナル（ネイティブビュー / ADR-0031）だけ不透明に残りうる。
  実機確認の結果次第で SwiftTerm 側の背景アルファ対応を別途検討する（D14）。
